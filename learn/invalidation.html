Invalidation

If your mom tells you dinner is at 6, you write it on a sticky note. If plans change to 7, you don't wait for the sticky note to fall off the fridge by itself — you take it down and fix it right now. That's invalidation.

🧒 Setting eviction aside for a moment

To keep this lesson focused, we'll go back to the plain object cache from Lesson 3, without the size limit from Lesson 5. Lesson 8 brings everything back together.

The problem TTL doesn't fully solve

In Lesson 4, we gave every cached answer a shelf life — after, say, 10 seconds, it's treated as expired. That handles data that goes stale gradually. But what about the moment you know for a fact the cached answer is now wrong — say, an admin manually corrects a reading? With only a TTL, everyone could keep seeing the old, wrong value for up to the rest of that 10-second window, even though you already know better.

The idea: clear the drawer the instant you know it's wrong

Instead of waiting for time to pass, we reach into the cache ourselves and remove the entry the moment the real data changes — at the exact same time we update the real data.

Building it, step by step

1

Add a way to correct the data

Let's add an endpoint that lets an admin manually fix a city's temperature:

server.js
app.put("/weather/:city", async (req, res) => {
  const city = req.params.city;
  const units = req.query.units || "celsius";
  const temperature = Number(req.query.temperature);

  saveWeatherToDatabase(city, temperature, units); // pretend this updates the real record
  // ...
});
2

Forget the cached answer for this exact key

server.js
const cacheKey = `${city}:${units}`;
delete cache[cacheKey];

res.json({ status: "updated" });

Unlike Python's del, JavaScript's delete operator never complains if the key isn't there — deleting something that doesn't exist is simply a no-op. So there's no need to check cacheKey in cache first; it's always safe to call directly.

The full file

server.js
const cache = {};

app.get("/weather/:city", async (req, res) => {
  const city = req.params.city;
  const units = req.query.units || "celsius";
  const cacheKey = `${city}:${units}`;

  if (cacheKey in cache) {
    return res.json(cache[cacheKey]);
  }

  await sleep(3000);
  const temperature = units === "celsius" ? 24 : 75;
  const weather = { city, temperature, units };
  cache[cacheKey] = weather;
  res.json(weather);
});

app.put("/weather/:city", async (req, res) => {
  const city = req.params.city;
  const units = req.query.units || "celsius";
  const temperature = Number(req.query.temperature);

  saveWeatherToDatabase(city, temperature, units);

  const cacheKey = `${city}:${units}`;
  delete cache[cacheKey];

  res.json({ status: "updated" });
});

Try it yourself

Terminal
curl http://localhost:3000/weather/paris        # ~3s, temperature: 24 (miss, then stored)
curl http://localhost:3000/weather/paris        # instant, temperature: 24 (hit)

curl -X PUT "http://localhost:3000/weather/paris?temperature=30"
# {"status": "updated"} — and the drawer for paris:celsius is now gone

curl http://localhost:3000/weather/paris        # ~3s again — cache miss, fetches the corrected value

The correction takes effect on the very next read — no waiting around for a TTL to run out.

Delete vs. update

We chose to delete the entry and let the next read repopulate it. An alternative is to immediately overwrite the cache with the corrected value, right there in the same endpoint:

alternative to `delete`
cache[cacheKey] = { city, temperature, units };

Deleting is simpler and always safe. Overwriting avoids one extra slow "miss" on the very next read, at the cost of having to carefully reconstruct the exact same shape the read endpoint expects — get that shape wrong and you'll cache a broken value instead of a missing one.

Recap

  • Invalidation means deliberately removing (or refreshing) a cached value the moment you know it's outdated, instead of waiting for a TTL.
  • The most common trigger is "whenever the real data is written, touch the cache too."
  • delete cache[key] is the simplest form of invalidation — clear it and let the next read rebuild it.
  • Always target invalidation using the same key-building logic as the read path, or you'll clear the wrong drawer (or none at all).
