Cache Keys

In the last lesson, every drawer in our box was labeled with just the city name. That works — until we ask the cache more than one kind of question. Then the label starts lying to us.

A bug hiding in plain sight

Let's make our weather endpoint more useful by letting people choose Celsius or Fahrenheit:

main.py
cache = {}

@app.get("/weather/{city}")
def get_weather(city: str, units: str = "celsius"):
    if city in cache:
        return cache[city]

    time.sleep(3)
    temperature = 24 if units == "celsius" else 75
    weather = {"city": city, "temperature": temperature, "units": units}
    cache[city] = weather
    return weather

Now watch what happens with two requests, back to back:

Terminal
curl "http://localhost:8000/weather/paris?units=celsius"
# {"city":"paris","temperature":24,"units":"celsius"}   ✅ correct

curl "http://localhost:8000/weather/paris?units=fahrenheit"
# {"city":"paris","temperature":24,"units":"celsius"}   ❌ wrong! still celsius
🧒 Why this happens

Our drawer is labeled only "paris" — it doesn't say anything about units. The line if city in cache only ever checks the city, so the second request sees a drawer already labeled "paris" and grabs it, without noticing it asked for something different. It's like two kids sharing one locker labeled just "Paris" — whoever opens it first decides what's inside for both of them, even though they wanted different things.

The fix: put everything that matters into the key

The label needs to describe the whole question, not just part of it. We build one string that combines both city and units:

main.py
cache = {}

@app.get("/weather/{city}")
def get_weather(city: str, units: str = "celsius"):
    cache_key = f"{city}:{units}"

    if cache_key in cache:
        return cache[cache_key]

    time.sleep(3)
    temperature = 24 if units == "celsius" else 75
    weather = {"city": city, "temperature": temperature, "units": units}
    cache[cache_key] = weather
    return weather

Try the same two requests again: each units value gets its own drawer, so each gets its own correct, freshly-labeled answer.

The rule to remember

📖 Rule

Anything that can change the answer must be part of the key. If it isn't in the key, it's invisible to the cache — and invisible bugs are the worst kind, because the code runs perfectly and still returns the wrong thing.

A few real examples of data that changes the answer, and so belongs in the key:

  • Per-user data — a "my orders" endpoint needs the user's ID in the key, or one user's cached order list could leak to another user.
  • Pagination — page 1 and page 2 of search results are different answers; the page number belongs in the key.
  • Language / locale — an English response and a French response for "the same" request are not the same cached value.

Don't overdo it, either

The opposite mistake is putting too much into the key — like including the exact current timestamp, or a random request ID. If every request produces a unique key, every request is a cache miss, and you've built a cache that never actually catches anything.

💡 Good keys only vary by what actually changes the answer

Two requests that would get the exact same correct answer should always produce the exact same key. Two requests that should get different answers must always produce different keys. Nothing more, nothing less.

Recap

  • A cache key is the label the cache uses to find (or store) an answer.
  • Every input that changes the correct answer must be part of the key.
  • An f-string like f"{city}:{units}" is a simple, common way to combine several values into one key.
  • Too little in the key causes wrong answers; too much in the key causes the cache to never hit.
