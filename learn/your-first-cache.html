Your First Cache

Let's build the smallest possible cache there is: a single JavaScript object. We'll go through every line of code — assuming nothing — and by the end you'll have a real endpoint that skips repeated slow work.

The problem, in code

Imagine we're building a weather app. Whenever someone asks for the weather in a city, our server calls a weather service — and pretend that service is slow, taking 3 whole seconds to answer. Here's that endpoint:

server.js
const express = require("express");
const app = express();

function sleep(ms) {
  return new Promise((resolve) => setTimeout(resolve, ms));
}

app.get("/weather/:city", async (req, res) => {
  const city = req.params.city;
  await sleep(3000); // pretend this is a slow call to a weather service
  res.json({ city, temperature: 24 });
});

app.listen(3000);

A quick word on sleep: Node.js runs your whole server on a single thread, so a real blocking pause (like Python's time.sleep()) would freeze every request, not just this one. sleep here instead returns a Promise that resolves after the given number of milliseconds — awaiting it pauses only this one request's handler, while the server keeps serving everyone else. This is the standard way to fake (or wait out) a delay in Node without blocking anything.

Run this and visit it a few times — even for the exact same city, every single visit waits the full 3 seconds. The server never remembers anything. Let's give it a memory.

A cache is just a box with labeled drawers

Picture a box of little drawers. Each drawer has a label — like "Paris" or "Tokyo" — and inside sits a piece of paper with the answer. JavaScript already has exactly this kind of box built in: a plain object.

a plain JS object, no Express yet
const box = {};             // an empty box, no drawers yet
box["Paris"] = 24;          // put 24 in a new drawer labeled "Paris"
console.log(box["Paris"]);  // open the "Paris" drawer and read what's inside -> 24

One more trick we'll use constantly: "Paris" in box asks "does a drawer with this label exist?" — without needing to open it. That's exactly what a cache needs to do first: check if an answer is already sitting in a drawer before doing any slow work.

Making our endpoint use the box

1

Create the box — but only once

We add one line, outside the request handler, near the top of the file:

server.js
const cache = {};
🧒 Why does it matter where this line lives?

If we wrote const cache = {} inside the request handler, JavaScript would build a brand-new, empty box every single time someone visited — throwing away whatever we'd saved a moment earlier. Writing it outside the handler means it's created once, when the server starts up, and it quietly stays alive in memory for as long as the server keeps running.

2

Check the box first — a cache hit

Before doing any slow work, look in the box:

server.js
if (city in cache) {
  return res.json(cache[city]);
}

If a drawer with this city's name already exists, we skip everything else and hand back what's inside immediately. This is exactly what a cache hit means. The extra return matters here too — without it, the code below would keep running even after we'd already sent a response.

3

If it's not there, do the slow work — then save it

This is a cache miss: the drawer was empty, so we do the real work and fill it in before we leave, so next time is a hit.

server.js
await sleep(3000);
const weather = { city, temperature: 24 };
cache[city] = weather;
res.json(weather);

The full endpoint

Put together, here's the whole file:

server.js
const express = require("express");
const app = express();

function sleep(ms) {
  return new Promise((resolve) => setTimeout(resolve, ms));
}

const cache = {};

app.get("/weather/:city", async (req, res) => {
  const city = req.params.city;

  if (city in cache) {
    return res.json(cache[city]);
  }

  await sleep(3000);
  const weather = { city, temperature: 24 };
  cache[city] = weather;
  res.json(weather);
});

app.listen(3000);

Try it yourself

Start the server (node server.js) and request the same city twice:

Terminal
curl http://localhost:3000/weather/paris
# ~3 seconds later: {"city":"paris","temperature":24}

curl http://localhost:3000/weather/paris
# instantly: {"city":"paris","temperature":24}

The second call comes back immediately. That's your cache working.

What did we just build?

No matter how many times someone asks for the same city, the slow work happens only once. Check first, do the work and save it only if you have to — this exact shape is something you'll see again and again in caching, even though we're not naming it yet.

⚠️ What this cache doesn't do yet
  • It treats every different input as a totally separate drawer — but what if two different questions should share an answer, or one question needs more than just a city to be correct? → Cache Keys
  • It never forgets. If the real weather changes, we keep serving today's number forever. → TTL
  • It has no size limit — cache a million cities and it grows forever. → Eviction
  • If someone corrects the data by hand, the cache doesn't know. → Invalidation

The next four lessons fix each of these, one small piece at a time.
