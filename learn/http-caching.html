HTTP Caching

Every lesson so far built a cache that lives inside our own server's memory. There's a completely different kind of caching that doesn't need our server at all — it happens on the visitor's own browser.

A cache with no drawers on our side at all

Imagine, instead of remembering the answer yourself, you write it on a piece of paper and hand it to your friend, saying: "Here — keep this for 30 seconds. Don't bother asking me again until then." Now your friend doesn't need to come back to you at all for a while. That's the idea behind HTTP caching: your server attaches a little note to its response, and the browser (or anything sitting in between, like a CDN) does the remembering.

The note is called a header: Cache-Control

Every HTTP response carries some extra information alongside the actual data — these are called headers. One of them, Cache-Control, is a standardized way of saying "here's how long you're allowed to reuse this response yourself, without asking me again."

Adding it to our endpoint

1

Ask FastAPI for the response object

main.py
from fastapi import FastAPI, Response

@app.get("/weather/{city}")
def get_weather(city: str, response: Response):
    ...

When you add a parameter typed as Response, FastAPI notices and hands you the outgoing response object to customize — including its headers — before it's sent.

2

Set the header

main.py
response.headers["Cache-Control"] = "public, max-age=30"
return weather

response.headers works just like a dictionary of little notes attached to the response. We're setting one note, made of two comma-separated parts: public (anyone is allowed to reuse this response, not just the one browser that asked — this matters for shared caches sitting in between, like a CDN) and max-age=30 (this note is good for 30 seconds, counting from right now).

The full endpoint

Notice this version has no internal Python dict at all — HTTP caching is a genuinely separate mechanism from everything in the earlier lessons, and can be used on its own:

main.py
import time
from fastapi import FastAPI, Response

app = FastAPI()

@app.get("/weather/{city}")
def get_weather(city: str, response: Response):
    time.sleep(3)  # our server still does the slow work every time
    weather = {"city": city, "temperature": 24}
    response.headers["Cache-Control"] = "public, max-age=30"
    return weather
🧒 Wait — isn't the server still slow?

Yes, on our side, every request still takes 3 seconds — this example has no server-side cache. But a real browser that receives this header won't even make the network request again for 30 seconds; it just reuses the copy it already has. Our server's speed didn't change, but the visitor's experience did.

Try it yourself

Terminal
curl -i http://localhost:8000/weather/paris

Look for this line in the response:

response headers
HTTP/1.1 200 OK
cache-control: public, max-age=30
content-type: application/json

{"city":"paris","temperature":24}

curl itself doesn't act on this header — it isn't a browser, so it always makes a fresh request. Open your browser's dev tools (Network tab) and reload the same URL within 30 seconds instead: you'll see the second request served (disk cache) or (memory cache), with no real network round-trip at all.

💡 There's a bit more to HTTP caching

Headers like ETag and Last-Modified let a browser ask "has this changed since I last checked?" and get back a tiny 304 Not Modified instead of the full response if it hasn't. Good to know it exists — not something you need in order to move on.

Recap

  • HTTP caching happens outside your server entirely — in the browser, or in something like a CDN sitting in between.
  • The Cache-Control response header is the note that controls it.
  • public, max-age=30 means "anyone may reuse this for 30 seconds."
  • It's independent of any in-memory cache your code keeps — you can use either, or both together.
