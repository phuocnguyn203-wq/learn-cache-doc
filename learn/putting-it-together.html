Putting It All Together

Seven lessons, seven small pieces, each set aside on its own so it was easy to see clearly. Now let's combine every piece into one complete, realistic file.

The complete file

main.py
import time
from collections import OrderedDict
from fastapi import FastAPI, Response

app = FastAPI()

cache = OrderedDict()
TTL_SECONDS = 30
MAX_ITEMS = 1000


def make_cache_key(city: str, units: str) -> str:
    return f"{city}:{units}"


@app.get("/weather/{city}")
def get_weather(city: str, response: Response, units: str = "celsius"):
    cache_key = make_cache_key(city, units)
    entry = cache.get(cache_key)

    if entry:
        age = time.time() - entry["saved_at"]
        if age < TTL_SECONDS:
            cache.move_to_end(cache_key)
            response.headers["Cache-Control"] = "public, max-age=30"
            return entry["data"]

    time.sleep(3)  # the slow work: calling the "real" weather service
    temperature = 24 if units == "celsius" else 75
    weather = {"city": city, "temperature": temperature, "units": units}

    if len(cache) >= MAX_ITEMS:
        cache.popitem(last=False)
    cache[cache_key] = {"data": weather, "saved_at": time.time()}

    response.headers["Cache-Control"] = "public, max-age=30"
    return weather


@app.put("/weather/{city}")
def correct_weather(city: str, temperature: float, units: str = "celsius"):
    save_weather_to_database(city, temperature, units)  # update the real record

    cache_key = make_cache_key(city, units)
    if cache_key in cache:
        del cache[cache_key]

    return {"status": "updated"}
💡 One small addition: make_cache_key

We pulled the key-building logic into its own tiny function. Both endpoints call it, so the GET and PUT paths can never accidentally build the key two different ways — which, as Lesson 6 warned, would mean invalidation silently clears the wrong drawer (or none at all).

Try the whole thing

Terminal
curl http://localhost:8000/weather/paris                          # ~3s — miss, cached
curl http://localhost:8000/weather/paris                          # instant — hit

curl -X PUT "http://localhost:8000/weather/paris?temperature=30"  # correction — cache cleared
curl http://localhost:8000/weather/paris                          # ~3s again — fresh, corrected value

curl -i http://localhost:8000/weather/paris                       # notice: Cache-Control header present too

What you now understand

Everything on this page is built from six ideas, each one small enough to hold in your head at once:

  • A cache is a box you check before doing slow work, and fill in after — what is caching
  • The key is the label the box uses to tell one question from another
  • A TTL puts an expiration date on a cached answer
  • Eviction keeps the box from growing forever when it's full
  • Invalidation clears an answer the instant you know it's wrong, without waiting
  • HTTP caching lets the browser itself skip asking your server at all
🧭 One honest question left

Everything here lives inside one server process. If you ran two or three copies of this app behind a load balancer — completely normal for real traffic — each copy would keep its own separate cache, with its own drawers. Solving that means reaching for a cache that lives outside any single process, shared by every copy of your app — tools like Redis are built for exactly that. It's a genuinely different topic worth learning on its own, once everything on this page feels natural.
