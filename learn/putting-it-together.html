Putting It All Together

Seven lessons, seven small pieces, each set aside on its own so it was easy to see clearly. Now let's combine every piece into one complete, realistic file.

The complete file

server.js
const express = require("express");
const app = express();

function sleep(ms) {
  return new Promise((resolve) => setTimeout(resolve, ms));
}

const cache = new Map();
const TTL_MS = 30000;
const MAX_ITEMS = 1000;

function makeCacheKey(city, units) {
  return `${city}:${units}`;
}

app.get("/weather/:city", async (req, res) => {
  const city = req.params.city;
  const units = req.query.units || "celsius";
  const cacheKey = makeCacheKey(city, units);
  const entry = cache.get(cacheKey);

  if (entry) {
    const age = Date.now() - entry.savedAt;
    if (age < TTL_MS) {
      cache.delete(cacheKey);
      cache.set(cacheKey, entry); // move to the "most recently used" end
      res.set("Cache-Control", "public, max-age=30");
      return res.json(entry.data);
    }
  }

  await sleep(3000); // the slow work: calling the "real" weather service
  const temperature = units === "celsius" ? 24 : 75;
  const weather = { city, temperature, units };

  if (cache.size >= MAX_ITEMS) {
    const oldestKey = cache.keys().next().value;
    cache.delete(oldestKey);
  }
  cache.set(cacheKey, { data: weather, savedAt: Date.now() });

  res.set("Cache-Control", "public, max-age=30");
  res.json(weather);
});

app.put("/weather/:city", async (req, res) => {
  const city = req.params.city;
  const units = req.query.units || "celsius";
  const temperature = Number(req.query.temperature);

  saveWeatherToDatabase(city, temperature, units); // update the real record

  const cacheKey = makeCacheKey(city, units);
  cache.delete(cacheKey);

  res.json({ status: "updated" });
});

app.listen(3000);
💡 One small addition: makeCacheKey

We pulled the key-building logic into its own tiny function. Both endpoints call it, so the GET and PUT paths can never accidentally build the key two different ways — which, as Lesson 6 warned, would mean invalidation silently clears the wrong drawer (or none at all).

Try the whole thing

Terminal
curl http://localhost:3000/weather/paris                          # ~3s — miss, cached
curl http://localhost:3000/weather/paris                          # instant — hit

curl -X PUT "http://localhost:3000/weather/paris?temperature=30"  # correction — cache cleared
curl http://localhost:3000/weather/paris                          # ~3s again — fresh, corrected value

curl -i http://localhost:3000/weather/paris                       # notice: Cache-Control header present too

What you now understand

Everything on this page is built from six ideas, each one small enough to hold in your head at once:

  • A cache is a box you check before doing slow work, and fill in after — what is caching
  • The key is the label the box uses to tell one question from another
  • A TTL puts an expiration date on a cached answer
  • Eviction keeps the box from growing forever when it's full
  • Invalidation clears an answer the instant you know it's wrong, without waiting
  • HTTP caching lets the browser itself skip asking your server at all
🧭 One honest question left

Everything here lives inside one server process. If you ran two or three copies of this app behind a load balancer — completely normal for real traffic — each copy would keep its own separate cache, with its own drawers. Solving that means reaching for a cache that lives outside any single process, shared by every copy of your app — Redis is built for exactly that, and it's a genuinely different topic worth learning on its own, now that everything on this page feels natural.
