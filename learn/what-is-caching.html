What is Caching?

Before touching a single line of code, let's build the right mental model. Everything in the basic and advanced tutorials is just a variation on the idea explained here.

The problem: doing the same work over and over

Imagine every time someone asked you what 17 × 24 was, you sat down and multiplied it out by hand — even if you'd been asked that exact question five seconds ago. That would be silly. You'd just remember the answer was 408 and say it immediately.

Web applications have this exact problem. A typical API endpoint might:

  • Run a slow database query joining several large tables
  • Call a third-party API that takes 2 full seconds to respond (and might charge you per call)
  • Perform a heavy computation — resizing an image, generating a report, running a recommendation model

If ten thousand people request the same product page in the same minute, does your server really need to redo that expensive work ten thousand times, to produce the exact same answer ten thousand times? Almost always, no.

📖 Definition

Caching is the practice of storing a copy of the result of some expensive work, so that the next time it's needed, you can hand back the stored copy instead of redoing the work.

The storage location itself — the thing holding the copy — is called a cache.

You already use caches every day

Caching isn't an exotic backend trick — it's something you rely on constantly without noticing:

🧠

Memorizing an answer

You don't recompute 17 × 24 every time — you remember "408" once you've worked it out. That memory is a cache.

📚

A book on your desk

Instead of walking to the library every time you need to check a fact, you keep the book on your desk until you're done with it.

🖼️

Your browser

Revisit a site and the logo appears instantly — your browser already downloaded that image and kept a copy on your disk.

Every one of these follows the same shape: do the work once, store the result, reuse it until it's no longer good enough.

Vocabulary you'll keep seeing

These six terms cover almost everything you'll read about caching. Skim them now — they'll click properly once you see them in code in the next tutorial.

Cache hit
The data you asked for was already in the cache. Fast path — no expensive work needed.
Cache miss
The data wasn't in the cache. You have to do the expensive work, and (usually) store the result for next time.
TTL (time-to-live)
How long a cached value is considered valid before it expires and must be refreshed. A price feed might have a TTL of 5 seconds; a country list might have a TTL of a day.
Eviction
Caches don't have infinite room. When they fill up, they need to throw something out to make space — that's eviction, and the rule for choosing what to throw out is an eviction policy (e.g. remove the Least Recently Used item first).
Invalidation
Deliberately removing or updating a cached value because you know it's now wrong — usually because the underlying data changed. This is the famously hard part of caching.
Stale data
A cached value that's still present but no longer matches reality, because it wasn't invalidated in time.
💬 The classic joke

"There are only two hard things in Computer Science: cache invalidation and naming things." You'll understand exactly why by the end of the advanced tutorial.

Caching happens at every layer

A single web request can pass through several caches before it ever reaches your database. As a web developer, you'll mostly work with the application-level cache, but it helps to see the whole picture:

🌐
Browser
Caches images, CSS, JS on the user's device
→
📡
CDN
Caches static content close to the user, worldwide
→
🚪
Reverse proxy
Nginx/gateway caches whole responses before they hit your app
→
🧩
Your app
In-memory or Redis cache inside FastAPI / Express
→
🗄️
Database
The source of truth — slowest, most expensive to hit

This guide focuses on the two layers you control directly as a backend developer: the application-level cache (in-memory or Redis, inside your FastAPI/Express process) and the HTTP cache headers that tell browsers and CDNs how to cache your responses. Those are exactly what the next two tutorials cover.

Seeing the effect: with vs. without a cache

Say an endpoint calls a slow upstream API that takes 800ms to respond, and it gets hit five times in a row for the same resource.

RequestWithout cachingWith caching
1st request800ms (does the work)800ms (cache miss — does the work, then stores it)
2nd request800ms (does the work again)~1ms (cache hit)
3rd request800ms~1ms (cache hit)
4th request800ms~1ms (cache hit)
5th request800ms~1ms (cache hit)

Without a cache: 4,000ms of total upstream work for 5 identical requests. With a cache: ~804ms — one real request, four nearly-instant ones. That gap only grows as traffic grows, which is exactly why caching matters more as your app scales.

When caching is the wrong tool

Caching isn't free, and it isn't always appropriate. Reach for it deliberately, not by default.

⚠️ Think twice before caching when...
  • The data changes on every request. A live "what time is it right now" endpoint has nothing worth caching.
  • Correctness matters more than speed. A bank balance shown right before a transfer should not be served from a 5-minute-old cache.
  • The data is per-user and unbounded. Caching by a key that has near-infinite unique values (e.g. a raw search string from every user) can grow your cache forever for little reuse benefit.
  • You haven't measured a real slowdown. Caching adds complexity — invalidation bugs, stale data, extra infrastructure. Don't pay that cost without a reason.

Recap

  • Caching = storing a result so you don't redo expensive work.
  • Key vocabulary: hit, miss, TTL, eviction, invalidation, stale.
  • Caching happens at many layers — browser, CDN, reverse proxy, your app, and sometimes the database itself.
  • It's a trade-off: faster responses in exchange for the risk of serving outdated data. Use it deliberately.
