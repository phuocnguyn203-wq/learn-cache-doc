Eviction

A backpack only has so much room. Once it's full, something has to come out before something new can go in. Caches face the exact same problem — and the rule for deciding what comes out is called an eviction policy.

🧒 Setting TTL aside for a moment

To keep this lesson focused on one idea, we'll temporarily drop the TTL code from the last lesson and go back to a simple hit/miss cache. Lesson 8 brings everything back together.

The problem: our cache can grow forever

Every new city-and-units combination gets its own drawer, and none of them are ever removed. If our weather app becomes popular and gets asked about thousands of different cities, our cache keeps growing, drawer after drawer, using more and more of the computer's memory — with no limit. Eventually, that can crash the server.

The idea: a size limit, plus a rule for what to remove

We need two things:

  1. A maximum number of drawers the cache is allowed to hold.
  2. A rule for choosing which drawer to empty out, once we're full and need room for one more.

The most common rule is called LRU — Least Recently Used. The idea, in plain words: the drawer nobody has touched in the longest time is the first one to go. It's the same instinct you'd use cleaning out a closet — the toy you haven't touched in months is the one you get rid of first, not the one you played with yesterday.

The tool: JavaScript's Map

To build this ourselves, we need a box that not only stores labeled drawers, but also remembers the order we last touched them, and lets us both peek at and remove the oldest one on demand. A plain {} object doesn't reliably guarantee order — but JavaScript's built-in Map does: it always iterates in the order keys were inserted, which is exactly the tool for the job.

server.js
const cache = new Map();
const MAX_ITEMS = 3;

Building it, step by step

1

On a hit, mark the drawer as "just touched"

server.js
if (cache.has(cacheKey)) {
  const value = cache.get(cacheKey);
  cache.delete(cacheKey);
  cache.set(cacheKey, value); // re-adding it moves it to the end
  return res.json(value);
}

Map has no built-in "move to end" method, so deleting a key and immediately setting it again is the standard trick — since a Map always iterates oldest insertion first, re-inserting it makes it the newest again. Without this, the cache would have no way of knowing this drawer was just used, and might evict it even though someone just asked for it.

2

On a miss, make room before adding something new

server.js
if (cache.size >= MAX_ITEMS) {
  const oldestKey = cache.keys().next().value; // remove the least recently used drawer
  cache.delete(oldestKey);
}

cache.keys() gives an iterator over the keys, oldest first — calling .next().value once grabs just that very first (oldest, least recently used) key without looping over anything.

The full endpoint

server.js
const cache = new Map();
const MAX_ITEMS = 3;

app.get("/weather/:city", async (req, res) => {
  const city = req.params.city;
  const units = req.query.units || "celsius";
  const cacheKey = `${city}:${units}`;

  if (cache.has(cacheKey)) {
    const value = cache.get(cacheKey);
    cache.delete(cacheKey);
    cache.set(cacheKey, value);
    return res.json(value);
  }

  if (cache.size >= MAX_ITEMS) {
    const oldestKey = cache.keys().next().value;
    cache.delete(oldestKey);
  }

  await sleep(3000);
  const temperature = units === "celsius" ? 24 : 75;
  const weather = { city, temperature, units };
  cache.set(cacheKey, weather);
  res.json(weather);
});

Watching it evict, step by step

With MAX_ITEMS = 3, here's what happens as requests come in:

RequestResultOrder after (oldest → newest)
GET /weather/parismiss, storedparis
GET /weather/tokyomiss, storedparis, tokyo
GET /weather/londonmiss, stored (now full)paris, tokyo, london
GET /weather/parishit — moved to the endtokyo, london, paris
GET /weather/berlinmiss — full, so tokyo is evicted (least recently used)london, paris, berlin
GET /weather/tokyomiss again — it was evicted, so it's slow once moreparis, berlin, tokyo

Notice that paris survived the eviction in the second-to-last row, purely because we happened to re-request it right before the cache filled up — that's the delete-and-re-insert trick doing its job.

💡 In real projects

You now know what's actually happening inside an LRU cache. In production code, most people reach for a tested library instead of hand-rolling this — the lru-cache npm package is the standard choice, and does exactly what we just built, with more edge cases handled. Understanding the mechanism first is what makes a tool like that make sense instead of feeling like magic.

Recap

  • Without a size limit, a cache can grow forever and eventually exhaust memory.
  • Eviction is the act of removing an entry to make room; the rule for choosing which one is the eviction policy.
  • LRU (Least Recently Used) evicts whichever entry has gone the longest without being accessed.
  • A Map's guaranteed insertion order, plus delete-then-set to "touch" an entry, is enough to hand-build a simple LRU cache.
