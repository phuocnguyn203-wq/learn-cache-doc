TTL & Expiration

Milk in the fridge has a date printed on the carton. You don't drink it forever just because it's sitting there. Cached data needs the same idea: a point where it stops being trustworthy.

The problem: our cache never forgets

In the last two lessons, once an answer goes into a drawer, it stays there forever. If the real temperature in Paris changes from 24° to 30° an hour later, our endpoint doesn't care — it keeps confidently returning 24°, because as far as the code knows, nothing ever expires.

The idea: remember when, not just what

Instead of only saving the answer, we'll also save a timestamp of the moment we saved it — like a sticky note that says "packed at 3:00 PM." Later, before trusting the drawer, we'll check how much time has passed.

JavaScript's built-in Date object gives us exactly the tool for this: Date.now().

just exploring, not part of our app yet
console.log(Date.now()); // e.g. 1737382021500 — "right now", as a single number
🧒 What is that number?

Date.now() returns the number of milliseconds since a fixed moment far in the past (midnight, January 1st, 1970 UTC — an arbitrary starting line computers agreed on decades ago). You never need to think about that date. All that matters: this number only ever goes up, so a later moment is always a bigger number than an earlier one. That means subtracting an old Date.now() from a new one tells you exactly how many milliseconds passed in between.

Building it, step by step

1

Store the answer and the timestamp together

Instead of putting the weather directly in the drawer, we put a small wrapper with two labeled compartments:

server.js
cache[cacheKey] = {
  data: weather,
  savedAt: Date.now(),
};

data holds the real answer; savedAt holds the timestamp from the moment we stored it.

2

When reading, check how old it is

server.js
const TTL_MS = 10000; // 10 seconds

const entry = cache[cacheKey];
if (entry) {
  const age = Date.now() - entry.savedAt;
  if (age < TTL_MS) {
    return res.json(entry.data);
  }
}

The full endpoint

server.js
const cache = {};
const TTL_MS = 10000;

app.get("/weather/:city", async (req, res) => {
  const city = req.params.city;
  const units = req.query.units || "celsius";
  const cacheKey = `${city}:${units}`;
  const entry = cache[cacheKey];

  if (entry) {
    const age = Date.now() - entry.savedAt;
    if (age < TTL_MS) {
      return res.json(entry.data);
    }
  }

  await sleep(3000);
  const temperature = units === "celsius" ? 24 : 75;
  const weather = { city, temperature, units };
  cache[cacheKey] = { data: weather, savedAt: Date.now() };
  res.json(weather);
});

Notice the shape hasn't really changed since Lesson 2: check first, do the slow work and save it if you have to. We've just made the "check" smarter — it now also asks "...and is it still fresh?"

Try it yourself

Terminal
curl http://localhost:3000/weather/paris   # ~3s (miss — nothing cached yet)
curl http://localhost:3000/weather/paris   # instant (hit — younger than 10s)

# wait 11 seconds...

curl http://localhost:3000/weather/paris   # ~3s again (miss — the entry expired)

Once more than 10 seconds pass, the drawer is still technically there — but we no longer trust it, so we redo the work and refresh it with a new timestamp.

How do you choose a TTL?

It's a trade-off, and it depends entirely on how fast the real answer changes and how outdated an answer is acceptable to show:

Kind of dataReasonable TTLWhy
Stock priceA few secondsChanges constantly; staleness is costly
WeatherA few minutesChanges slowly; a short delay is harmless
List of countriesHours or a dayBasically never changes

Shorter TTL → fresher data, but more slow work happening. Longer TTL → less slow work, but data can be wrong for longer. There's no universal right answer, only a trade-off to make deliberately.

Recap

  • A TTL (time-to-live) is how long a cached answer is trusted before it's treated as expired.
  • Date.now() gives you an always-increasing "right now" number in milliseconds; subtracting two of them gives elapsed time.
  • Storing { data: ..., savedAt: ... } instead of the raw answer lets us check freshness later.
  • Choosing a TTL is a trade-off between freshness and how much slow work you're willing to repeat.
