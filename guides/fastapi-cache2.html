Caching Libraries

Hand-writing "check the cache, load on a miss, populate it" at every endpoint gets repetitive fast. Both ecosystems have a real, widely-used library that turns this into a single line added to a route: fastapi-cache2 for FastAPI, and apicache for Express. Both are read-through caches (see Caching Patterns) — your handler doesn't know or care whether it hit or missed.

Installing

Terminal
pip install "fastapi-cache2[redis]"
Terminal
npm install apicache redis

Initializing

fastapi-cache2 requires an explicit backend before anything can be cached. apicache takes the opposite default: it caches in memory automatically, and only needs configuring if you want it backed by Redis instead.

main.py
from contextlib import asynccontextmanager
from fastapi import FastAPI
from fastapi_cache import FastAPICache
from fastapi_cache.backends.redis import RedisBackend
from redis import asyncio as aioredis

@asynccontextmanager
async def lifespan(app: FastAPI):
    redis = aioredis.from_url("redis://localhost:6379")
    FastAPICache.init(RedisBackend(redis), prefix="fastapi-cache")
    yield

app = FastAPI(lifespan=lifespan)
server.js
const express = require("express");
const apicache = require("apicache");
const { createClient } = require("redis");

const app = express();
const redisClient = createClient({ url: "redis://localhost:6379" });

async function start() {
  await redisClient.connect();
  apicache.options({ redisClient });   // omit this line to use apicache's in-memory store instead
  app.listen(3000);
}
start();

const cache = apicache.middleware;
⚠️ Know your library's default

Because apicache silently falls back to an in-memory store, a misconfigured or never-connected redisClient won't error — it'll just quietly cache per-process instead of per-fleet, reintroducing the exact multi-instance problem Redis Caching exists to solve. Verify the Redis connection explicitly if you're relying on it.

Caching a route

main.py
from fastapi_cache.decorator import cache

@app.get("/products/{product_id}")
@cache(expire=60)
async def read_product(product_id: int):
    return await db.fetch_product(product_id)
server.js
app.get("/products/:productId", cache("1 minute"), async (req, res) => {
  const product = await db.fetchProduct(req.params.productId);
  res.json(product);
});

Same idea, different extension mechanism per framework. FastAPI's @cache() is a decorator wrapping your function; Express's cache(...) is literal middleware sitting in the route's handler chain, ahead of your function — on a hit, it responds and your handler never runs at all. Both are read-through: neither leaves you a miss branch to write.

Customizing the cache key

Both libraries build a default key from the request (FastAPI: function + arguments; apicache: request URL). Neither default distinguishes per-user data by itself — exactly the cache key principle from Learn: anything that changes the correct answer must be part of the key.

main.py
from fastapi import Depends
from fastapi_cache.decorator import cache

def user_scoped_key_builder(func, namespace, request, response, *args, **kwargs):
    user_id = kwargs["current_user"].id
    return f"{namespace}:{func.__module__}:{func.__name__}:user:{user_id}"

@app.get("/me/orders")
@cache(expire=30, key_builder=user_scoped_key_builder)
async def get_my_orders(current_user: User = Depends(get_current_user)):
    return await db.fetch_orders_for(current_user.id)
server.js
apicache.options({
  appendKey: (req, res) => (req.user ? req.user.id : "anonymous"),
});

app.get("/me/orders", cache("30 seconds"), async (req, res) => {
  res.json(await db.fetchOrdersFor(req.user.id));
});

key_builder replaces FastAPI's default key entirely; apicache's appendKey instead extends its URL-based default with whatever you return — same principle, different shape of the API.

Manual invalidation

main.py
@app.put("/products/{product_id}")
async def update_product(product_id: int, data: ProductUpdate):
    await db.save_product(product_id, data)
    await FastAPICache.clear(namespace="read_product")
    return {"status": "updated"}
server.js
app.put("/products/:productId", async (req, res) => {
  await db.saveProduct(req.params.productId, req.body);
  apicache.clear(`/products/${req.params.productId}`);
  res.json({ status: "updated" });
});

FastAPICache.clear(namespace=...) defaults to the endpoint's function name, so it clears every cached product_id for that endpoint at once — simple, but blunt. apicache.clear(...) targets one specific cached URL, which happens to be more precise here — call it with no argument at all to clear everything instead.

When to reach for a library vs. rolling your own

Use a libraryRead-heavy endpoints you can cache as a whole response, with a uniform TTL and no need for fine-grained, per-key invalidation.
Roll your own (Redis Caching)You need precise per-key invalidation, per-request TTL logic, stampede protection, or caching only part of a response.

Recap

  • Both libraries implement read-through caching — your handler never sees a miss branch.
  • FastAPI wraps the function with a decorator; Express inserts middleware into the route's handler chain.
  • Override the default cache key whenever it wouldn't correctly distinguish two different requests — key_builder replaces it, appendKey extends it.
  • apicache silently defaults to in-memory storage; verify your Redis wiring explicitly if that's not what you want.
