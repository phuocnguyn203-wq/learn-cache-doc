Redis Concepts

Redis with Node.js — and really every Redis client library in every language — is a thin wrapper around the same underlying engine. This page is that engine, explained simply: what Redis actually is, the data structures it gives you beyond plain key-value, how it decides what to evict and when, whether your data survives a restart, and what "atomic" really guarantees. No prior Redis knowledge assumed — read this once and the client library stops feeling like a black box.

What is Redis, really?

Redis (REmote DIctionary Server) is an in-memory data structure store. Two words in that sentence matter more than they first appear:

⚡

In-memory

The entire dataset lives in RAM, not on disk. That's what makes reads and writes take microseconds instead of the milliseconds a disk-backed database needs. It's also why capacity planning for Redis means "how much RAM," not "how much disk."

🧩

Data structure store

A plain key-value store only gives you strings in, strings out. Redis gives every value a type — hash, list, set, sorted set, and more — each with its own commands, so the server does structural work (sorting, counting, deduplicating) your application would otherwise do after fetching a blob.

🧭 "Just a cache" undersells it

Redis is most commonly reached for as a cache, because that's the easiest entry point. In production, the same server is just as often a session store, a message broker (pub/sub or Streams), a rate limiter, a leaderboard, or a distributed lock — all built from the same handful of data structures below. Nothing about Redis itself privileges the "cache" use case; it's just the one most people learn first.

One more property shapes almost everything else on this page: Redis executes commands one at a time, in order, on a single thread. There's no risk of two commands interleaving mid-execution the way there is with a multi-threaded server touching shared memory. That single fact is why a command like INCR is safe under concurrency without you doing anything special — the "read old value, add one, write it back" never gets interrupted partway through. (Redis 6+ added optional multi-threaded I/O to move network reads/writes off the main thread under heavy load — but command execution itself is still strictly one-at-a-time. The atomicity guarantee doesn't change.)

The data structures

Forget Redis for a second and picture a plain JavaScript object, or a Python dict — one big map of key: value pairs, sitting in memory. Redis is that map, except it lives in a separate process that your application talks to over the network instead of holding directly in its own memory. That's the entire mental model. The one twist that makes Redis more than "a dict on another server" is that the value isn't limited to a plain string — each key's value has a declared type, and that type decides both what shape the value takes and which commands are legal against it. You can't LPUSH onto a key that holds a String, any more than you could call .push() on a plain number in JavaScript.

Here's what that looks like with five real keys sitting in one Redis server at once — this is the picture to hold in your head for everything below:

One Redis server, five keys, five shapes
"user:1000:name"    →  STRING  "Ada Lovelace"
"user:1000"         →  HASH    {name: "Ada", plan: "pro", logins: 42}
"recent:user:1000"  →  LIST    ["viewed:57", "viewed:42", "viewed:31"]
"product:42:tags"   →  SET     {"sale", "electronics", "new"}
"leaderboard"       →  ZSET    {"carol": 1200, "alice": 1500, "bob": 1800}

Every one of those keys is fetched, inspected, and modified with a different set of commands, because a list of recent views, a bag of unique tags, and a leaderboard all need different operations to be fast. That's really all "Redis has data structures" means: instead of forcing every problem into "fetch the whole blob, change it in your app, write the whole blob back," Redis gives the shape that matches your access pattern, with server-side commands built for that shape. The five below cover the overwhelming majority of real usage.

🧭 A note on the "O(1)" / "O(n)" you'll see below

Each type description below mentions how its operations scale, using Big-O notation: O(1) means "takes roughly the same amount of time no matter how much data is in there" (constant time — the best case). O(n) means "takes longer the more items there are" (linear — fetching 1,000 items takes roughly 10x as long as fetching 100). O(log n) sits in between: it grows, but much more slowly than the data does. It matters here because it tells you which operations stay cheap as your dataset grows, and which ones quietly become the bottleneck.

🔤

String

Binary-safe text or bytes, up to 512MB. The default type — a JSON blob, a serialized object, a plain number.

🗂️

Hash

A field → value map living under one key. A small object embedded directly in Redis, without a second key per field.

📋

List

An ordered sequence of strings, push/pop from either end in O(1). Queues and recent-activity feeds.

🎯

Set

Unordered, unique strings. Membership checks and set algebra — union, intersect, difference — all O(1) or close to it.

🏆

Sorted Set

Like a Set, but every member carries a floating-point score and stays ordered by it. Leaderboards, priority queues, rate-limit windows.

Strings

The simplest type — a single value under a key, like a labeled drawer holding one piece of paper. SET writes the whole value; there's no partial update — writing a new value always replaces the old one. If you've used Redis only as "the thing my cache library talks to," this is almost certainly the type it's been using under the hood.

Shape in memory
key                    value
────────────────────────────────────
"user:1000:name"  →  "Ada Lovelace"

one key, one opaque value — Redis doesn't know or care what's inside it

Walking through what each command actually does to that drawer, step by step:

redis-cli
SET user:1000:name "Ada"
GET user:1000:name                  # → "Ada"
APPEND user:1000:name " Lovelace"   # → "Ada Lovelace" (13 chars)
STRLEN user:1000:name               # → 13
INCR page:views                     # → 1 (key created fresh)
INCRBY page:views 10                # → 11

INCR/INCRBY only work because Redis treats a string holding a valid integer as a number for these commands — under the hood it parses the string, does the math, and writes the result back as a string again, all as one atomic step. That single fact is the whole basis for Redis-backed counters and rate limiters, covered in Redis with Node.js.

Hashes

A hash stores several named fields under one key — think of it as a mini object (in JavaScript terms, one key pointing at { name: "Ada", plan: "pro", logins: 42 }) living directly inside one drawer, instead of one piece of paper per drawer. Useful whenever you'd otherwise reach for several separate string keys that always change together (a user's profile fields, cached metadata alongside the cached value itself).

Shape in memory
"user:1000"  →  field      value
              ─────────────────
              name       "Ada"
              plan       "pro"
              logins     42

one key, many named fields — like a small table nested inside a single entry

HSET can create or update several fields in one call; HGET reads a single field back without touching the rest of the hash — that's the whole advantage over stuffing the same data into one JSON string, where reading one field means deserializing everything:

redis-cli
HSET user:1000 name "Ada" plan "pro" logins 42   # → 3 (new fields created)
HGET user:1000 plan                # → "pro"
HGETALL user:1000                  # → name Ada, plan pro, logins 42
HINCRBY user:1000 logins 1         # → 43
HDEL user:1000 plan                # → 1 (plan field removed)
💡 Hash vs. several strings vs. one JSON string

Three ways to store the same "user profile" data, three different trade-offs. Separate string keys (user:1000:name, user:1000:plan) let you expire or fetch one field independently, at the cost of one round trip per field. A hash gets you most fields in one key with per-field access (HGET) — but a hash has one TTL for the whole key, not per field. A single JSON string is the cheapest to write and the easiest to reason about, but you always pay to (de)serialize the entire blob even to read one field.

Lists

An ordered sequence of strings under one key — a doubly linked list under the hood, which is the detail that explains its whole performance profile. Pushing or popping at either end is O(1) (Redis just relinks the head or tail pointer — no matter if the list holds 10 items or 10 million, that's the same tiny amount of work). But reaching into the middle (LINDEX) is O(n), because there's no shortcut — Redis has to walk the chain from one end. That shape makes lists the natural fit for queues and bounded activity feeds, not for anything you'll binary-search or randomly index into.

Shape in memory
"recent:user:1000"  →  [ "viewed:57" | "viewed:42" | "viewed:31" ]
                       ↑head                      tail↑
                    LPUSH/LPOP                RPUSH/RPOP

cheap to change at either end, expensive to search in the middle

Walking through a typical "keep the 100 most recent events" pattern:

redis-cli
LPUSH recent:user:1000 "viewed:product:42"   # → 1 (list length)
LPUSH recent:user:1000 "viewed:product:57"   # → 2 (newest at head)
LRANGE recent:user:1000 0 9         # → up to 10 items, newest (head) first
LTRIM recent:user:1000 0 99         # keeps newest 100, drops the rest
RPOP job:queue                      # → tail item removed (FIFO queue)

LPUSH (add at head) + RPOP (remove from tail) together give you a first-in-first-out queue — items leave in the same order they arrived. LPUSH + LPOP (both at the head) gives a stack instead — last in, first out. BLPOP is the blocking variant of LPOP: a worker calls it and the connection just waits (up to a timeout) until an item shows up, instead of polling LPOP in a loop.

Sets

Unordered, unique members — add the same value twice and the set simply stays the same size. There's no index, no first or last element, and no guaranteed order at all: a Set answers "is this in there?" and "what's in there in common with that other one?", not "what's item #3?". The killer feature isn't storage — it's the set-algebra commands (intersect, union, diff) that run entirely server-side, so you never have to pull two potentially huge sets into your application just to compare them yourself.

Shape in memory
"product:42:tags"  →  { "sale", "electronics", "new" }

a bag with no order and no duplicates — adding "sale" again changes nothing
redis-cli
SADD product:42:tags "sale" "electronics" "new"   # → 3 (members added)
SISMEMBER product:42:tags "sale"    # → 1 (present), O(1) either way
SCARD product:42:tags               # → 3 (count of members)
SINTER product:42:tags product:57:tags   # → members present in BOTH sets
SUNION viewers:today viewers:yesterday   # → union, no duplicates

Sorted Sets (ZSets)

Take everything a Set gives you — unique members, fast membership checks — and add one thing: every member also carries a floating-point score, and Redis automatically keeps the whole set ordered by that score at all times (implemented internally as a skip list, a structure built specifically so inserting into the middle of a sorted order costs O(log n) instead of O(n)). You never sort a sorted set yourself — it's always already sorted, by construction, after every single write. That's the single most versatile structure in Redis: a leaderboard uses the score as points; a rate limiter or delayed job queue uses the exact same structure with a Unix timestamp as the score instead.

Shape in memory — always kept sorted by score
"leaderboard"  →  score   member
                    ─────────────
                    1200    "carol"
                    1500    "alice"
                    1800    "bob"

member = who; score = what it's sorted by (points, a timestamp, etc.)

Watching how each command reads or reshuffles that ordering:

redis-cli
ZADD leaderboard 1500 "alice" 1800 "bob" 1200 "carol"   # → 3 (members added)
ZREVRANGE leaderboard 0 2 WITHSCORES   # → bob, alice, carol (high to low)
ZSCORE leaderboard "alice"              # → "1500"
ZINCRBY leaderboard 50 "alice"          # → "1550" (new score)
ZRANK leaderboard "alice"               # → 1 (ascending, 0-indexed)
ZRANGEBYSCORE leaderboard 1000 1600     # → carol, alice (1000-1600)

Using a Unix timestamp as the score turns a sorted set into a time-ordered index — the basis for the sliding-window rate limiter shown in Redis with Node.js.

Beyond the core five

Redis ships several more specialized structures, all worth knowing exist even without going deep here:

TypeWhat it's for
StreamsAn append-only log of entries, each with an ID. Unlike pub/sub (below), messages persist and can be replayed — the building block for a Redis-backed message queue with consumer groups.
BitmapsBit-level operations on an ordinary string — compact boolean flags per user/day, like "did user N log in today," at roughly 1 bit per user.
HyperLogLogA probabilistic structure that estimates the count of unique items using a fixed ~12KB, regardless of how many items you add — trading exactness for constant memory on "how many unique visitors" style questions.
GeospatialBuilt on top of sorted sets: store lat/long points and query "what's within N km of here" directly in Redis.

Expiration: how TTL actually works

The Learn section covers TTL as a concept; here's the mechanism underneath. Redis doesn't watch a clock and delete keys the instant they expire — checking every key on every tick would be wasteful. Instead it combines two strategies:

  • Lazy expiration — every time a key is accessed, Redis checks its expiry first. If it's past due, Redis deletes it on the spot and treats the access as a miss.
  • Active expiration — a background cycle runs several times a second, sampling a small batch of keys that have a TTL set, deleting any that have expired. This catches keys that are never accessed again, so they don't sit in memory forever.

Between an EXPIRE call and the moment a key is actually gone, there's a small window where it's expired-but-not-yet-purged, which is invisible to clients — any read still correctly sees a miss.

redis-cli
EXPIRE session:abc123 3600      # expire in 3600 seconds
PEXPIRE session:abc123 3600000  # same thing, in milliseconds
EXPIREAT session:abc123 1735689600  # expire at a specific Unix timestamp
TTL session:abc123              # seconds remaining; -1 = no TTL, -2 = doesn't exist
PERSIST session:abc123          # remove the TTL — key now lives forever

Eviction: what happens when memory runs out

Redis is bounded by the maxmemory setting (if you set one). Once the dataset hits that ceiling, maxmemory-policy decides what happens next — and the default surprises a lot of people who assume Redis just quietly starts evicting.

PolicyBehavior at the memory limit
noeviction (default)Writes start failing with an error. Reads keep working. Nothing is ever evicted automatically.
allkeys-lruEvict the least-recently-used key, from the entire keyspace, regardless of whether it has a TTL.
allkeys-lfuEvict the least-frequently-used key across the whole keyspace — better than LRU when "used a lot a while ago" should outweigh "used once, very recently."
allkeys-randomEvict a random key from the entire keyspace.
volatile-lruEvict the least-recently-used key, but only among keys that have a TTL set. Keys without a TTL are never touched.
volatile-lfuSame idea as volatile-lru, using frequency instead of recency.
volatile-randomEvict a random key among keys that have a TTL set.
volatile-ttlEvict the key with the soonest expiration among keys that have a TTL — evicting early instead of waiting for it to expire naturally.
⚠️ The default will silently reject writes

If you never touch maxmemory-policy, it's noeviction. That's the safe choice for Redis-as-a-database, but for Redis-as-a-cache it means: hit the memory ceiling, and every SET across your entire app starts erroring — reads still succeed, so the failure mode is confusing ("why are writes failing but everything looks fine?"). If Redis is purely a cache in your architecture, volatile-lru (only evict keys that have a TTL — protects any non-expiring keys you rely on) or allkeys-lru (if every key is disposable) are almost always what you actually want.

One more detail worth knowing: Redis's "LRU" isn't a perfectly maintained least-recently-used queue — tracking exact recency for every key would cost memory and CPU on its own. Instead, when eviction is needed, Redis samples a small number of random keys (5 by default, tunable via maxmemory-samples) and evicts the least-recently-used of that sample. More samples means closer to true LRU, at a small extra CPU cost — the default is a deliberate, good-enough trade-off.

Persistence: does your data survive a restart?

"In-memory" doesn't have to mean "gone forever if the process dies." Redis offers two independent, combinable persistence mechanisms — and for a pure cache, the right answer is often to use neither.

RDBPoint-in-time snapshots of the whole dataset, written to a single compact file on a schedule (e.g. "if at least 100 keys changed in the last 5 minutes, snapshot") or on demand via BGSAVE. Fast to restart from, but anything written since the last snapshot is lost on a crash.
AOFAppend-only file — every write command is logged as it happens. Far more durable (configurable down to zero data loss), but the file grows continuously and needs periodic compaction (BGREWRITEAOF), and replaying a large AOF on startup is slower than loading an RDB snapshot.

AOF's durability is itself tunable via appendfsync: always (fsync to disk on every single write — safest, slowest), everysec (fsync once a second — the default and the standard middle ground: at most one second of writes lost on a crash), or no (let the OS decide when to flush — fastest, least durable). Many production deployments run both RDB and AOF together (a "hybrid" mode, where AOF rewrites start from an RDB-format preamble) for fast restarts and strong durability.

💡 For a pure cache, persistence is often a deliberate no

If Redis only ever holds derived data — cached query results, sessions that can be re-authenticated, rate-limit counters — your database (or the user re-logging in) is already the real source of truth. Losing the cache on a restart just means a wave of misses while it refills, not data loss. Plenty of caching-only Redis deployments run with both RDB and AOF disabled entirely, trading "survives a restart" for a simpler, slightly faster server. Turn persistence on the moment Redis starts holding anything that isn't reconstructible elsewhere.

Atomicity: single commands, transactions, and scripts

Single-threaded execution (from the top of this page) means every individual Redis command is already atomic — no other command can run partway through it. The question is what happens when your logic needs more than one command to stay atomic together.

MULTI / EXEC: transactions

Commands issued after MULTI are queued, not executed, until EXEC runs them all as one uninterruptible block — no other client's commands can interleave between them.

redis-cli
MULTI
INCR page:views
EXPIRE page:views 86400
EXEC              # both commands run together, atomically

This is batching with a guarantee, not conditional logic — Redis doesn't support branching inside a transaction (there's no "if this, then that" between MULTI and EXEC). For read-then-write logic that needs to stay consistent, pair it with WATCH.

WATCH: optimistic locking

WATCH key tells Redis "abort my next transaction if this key changes before I call EXEC." It's the standard way to do a safe read-modify-write — check a value, decide what to write based on it, and guarantee nobody else changed it out from under you in between.

redis-cli
WATCH inventory:sku42
GET inventory:sku42        # read the current value in your app code
MULTI
DECR inventory:sku42
EXEC                       # returns (nil) if inventory:sku42 changed since WATCH

If EXEC returns nil, nothing was written — your application is expected to retry the whole read-decide-write cycle from the top. This is compare-and-swap, not a lock: nobody is blocked waiting, contending clients just retry.

Lua scripting: server-side logic

EVAL runs a Lua script inside Redis, on the same single thread as every other command — the whole script executes atomically, and it can contain real conditional logic that MULTI/EXEC can't. It's the tool of choice when a transaction's "no branching" limitation is actually the problem — like the standard safe pattern for releasing a distributed lock (covered in Redis with Node.js), which needs to check and delete atomically only if a value still matches.

Pub/Sub: fire-and-forget messaging

PUBLISH sends a message to a channel; every client currently subscribed via SUBSCRIBE receives it immediately. It's genuinely fire-and-forget: Redis doesn't store the message anywhere, and a subscriber that wasn't connected at the moment of publishing simply never sees it — there's no history, no replay, no queue.

redis-cli
SUBSCRIBE cache-invalidations
> in another connection:
PUBLISH cache-invalidations "product:42"

That's exactly the right fit for something like broadcasting a cache invalidation to every server instance (see Invalidation at Scale) — if a server is briefly down, it'll rebuild its local cache from Redis or the database anyway, so a missed invalidation message isn't a correctness problem. It's the wrong fit for anything that must not be lost — a job queue, an event log, a billing event. For that, reach for Streams instead: durable, replayable, and built for consumer groups that track their own read position.

When Redis isn't the right tool

⚠️ Keep these limits in mind
  • Your dataset must fit in RAM. RAM is expensive relative to disk — Redis isn't the place for your entire multi-terabyte analytics warehouse.
  • It's not free of network cost. Every command is still a round trip. Fetching one key is fast; fetching ten thousand keys one at a time in a loop is a self-inflicted bottleneck — batch with MGET/pipelining instead.
  • Durability takes deliberate configuration. With persistence off (or set to lossy AOF settings), a crash can lose data. Fine for a cache; not fine if Redis is quietly becoming your system of record.
  • It's not a relational database. No joins, no ad-hoc queries across arbitrary fields, no foreign keys. Model your access patterns into keys up front, the way you'd design an index, not a schema.

Recap

  • Redis is an in-memory data structure store — strings, hashes, lists, sets, and sorted sets each have their own purpose-built commands, not just generic get/set.
  • Single-threaded command execution is what makes individual commands atomic; MULTI/EXEC/WATCH extend that guarantee across several commands, and Lua scripts extend it further to include branching logic.
  • Expiration is lazy (checked on access) plus active (a background sweep) — never an instant, guaranteed-the-moment-it-happens delete.
  • The default eviction policy, noeviction, makes writes start failing at the memory limit rather than silently evicting — pick a policy deliberately for a cache workload.
  • Persistence (RDB and/or AOF) is a durability-vs-performance dial, and it's entirely reasonable to leave it off for data that's fully disposable.
  • Pub/sub is fire-and-forget with no history; reach for Streams when messages must not be lost.
