Invalidation at Scale

Invalidation in Learn clears one key in one process. Two things get harder once you're running a fleet: propagating invalidation to caches that aren't Redis itself, and avoiding a stampede when a hot key does expire.

Two-tier caching and why Pub/Sub matters

A common design layers a small local (in-process) cache in front of Redis, to avoid even the network round-trip for extremely hot keys. Deleting a key from Redis clears the shared layer instantly — but every instance's local copy is untouched until its own TTL happens to expire. You need a way to tell every instance, right now, "forget this key."

Redis Pub/Sub solves exactly this: broadcast an invalidation message, and every instance listening clears its own local entry.

server.js
const localCache = new Map();

async function invalidateProduct(productId, r) {
  const cacheKey = `product:${productId}`;
  await r.del(cacheKey);
  await r.publish("cache-invalidation", cacheKey);
}

async function listenForInvalidations(r) {
  const subscriber = r.duplicate();
  await subscriber.connect();
  await subscriber.subscribe("cache-invalidation", (cacheKey) => {
    localCache.delete(cacheKey);
  });
}

subscribe() takes a callback directly, invoked once per message — no manual message-loop to write. Note also r.duplicate(): a Redis connection that's subscribed to a channel can't issue ordinary commands on that same connection, so it needs its own dedicated connection, separate from the one your app uses for everything else. See Redis if duplicate() is new to you.

listenForInvalidations runs once per instance, started right after your main Redis client connects. When any instance calls the invalidate function, every instance's local cache drops that key within milliseconds — regardless of which instance served the original write.

The cache stampede problem

When a popular key expires, many concurrent requests can miss at the exact same instant and all recompute the same value simultaneously — multiplying load on the database by however many requests happened to arrive in that window. This is independent of whether you're on one instance or many; a shared Redis cache doesn't prevent it by itself.

Fix 1: a distributed lock

Let only the first request that sees a miss recompute; everyone else waits briefly and retries.

server.js
async function getProductLocked(productId, r) {
  const cacheKey = `product:${productId}`;
  const lockKey = `lock:${cacheKey}`;

  const cached = await r.get(cacheKey);
  if (cached) return JSON.parse(cached);

  const gotLock = await r.set(lockKey, "1", { NX: true, EX: 5 });
  if (gotLock) {
    try {
      const product = await db.fetchProduct(productId);
      await r.set(cacheKey, JSON.stringify(product), { EX: 300 });
      return product;
    } finally {
      await r.del(lockKey);
    }
  }

  await new Promise((resolve) => setTimeout(resolve, 50));
  return getProductLocked(productId, r);
}

SET ... NX EX is atomic in Redis — exactly one caller can hold the lock at a time, and it self-expires even if the holder crashes before releasing it. set(..., { NX: true }) returns a truthy value only when the key didn't already exist and the write succeeded.

Fix 2: probabilistic early expiration (XFetch)

Locking prevents the stampede but still makes every other request wait. A more sophisticated approach — described in Vattani, Chierichetti & Lowenstein's "Optimal Probabilistic Cache Stampede Prevention" and used in production at Facebook — avoids the hard cliff entirely: instead of "valid until exactly T, then everyone recomputes," each read has a small, rising probability of recomputing early, well before the real expiry. A few unlucky requests refresh the value ahead of time, so by the time it would have actually expired, it never does — no simultaneous miss, no stampede.

server.js
async function getProductXfetch(productId, r, beta = 1.0) {
  const cacheKey = `product:${productId}`;
  const entry = await r.hGetAll(cacheKey); // {} if the key doesn't exist

  if (entry.expiry) {
    const expiry = Number(entry.expiry);
    const delta = Number(entry.delta); // how long the last computation took
    const now = Date.now() / 1000;
    if (now - delta * beta * Math.log(Math.random()) < expiry) {
      return JSON.parse(entry.data);
    }
  }

  const start = Date.now() / 1000;
  const product = await db.fetchProduct(productId);
  const delta = Date.now() / 1000 - start;

  await r.hSet(cacheKey, {
    data: JSON.stringify(product),
    expiry: String(start + 300),
    delta: String(delta),
  });
  return product;
}
💡 Why check entry.expiry instead of just entry?

In JavaScript, every object is truthy — even an empty one, {}, which is exactly what hGetAll returns for a key that doesn't exist. So if (entry) would incorrectly pass even on a total miss. Checking a specific field that only exists on a real entry, like entry.expiry, correctly evaluates to undefined (falsy) when there's nothing cached yet.

-delta * beta * log(random()) is near zero right after caching (so almost every read is a normal hit) and grows as now approaches expiry — smoothly increasing the chance of an early, single-request recompute the closer you get to real expiration. delta means slower-to-compute values start renewing earlier, since a stampede on an expensive query is more costly than one on a cheap one.

Locking vs. XFetch

LockingSimple to reason about; guarantees exactly one recompute at a time; adds latency for every caller that has to wait, and requires tuning the lock's own expiry.
XFetchNo caller ever blocks; recomputation spreads out smoothly instead of concentrating; more moving parts (extra stored metadata, a formula to get right) and a small chance of very slightly early recomputation.

Recap

  • Redis Pub/Sub propagates invalidation to local, per-instance caches that a plain DEL on Redis can't reach — subscribe with a callback, on a dedicated duplicate()d connection.
  • A cache stampede is many requests recomputing the same expired key at the same instant — it gets worse, not better, just because you're using a shared cache.
  • A distributed lock (SET NX EX) guarantees only one recompute, at the cost of making other callers wait briefly.
  • Probabilistic early expiration (XFetch) avoids blocking entirely by spreading recomputation out before the real expiry.
