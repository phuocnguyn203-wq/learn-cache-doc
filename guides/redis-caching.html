Redis Caching

A single in-process cache stops working the moment you run more than one instance of your app. Redis gives every instance a shared store — this guide covers wiring it into both FastAPI and Express, and goes through the actual Redis command API you'll use day to day, not just one client library's wrapper around it.

Installing

Terminal
pip install redis
Terminal
npm install redis

Both examples use each ecosystem's official client — redis-py (imported as redis.asyncio for the async API) and Node's official redis package (v4+, promise-based). Their method names differ in small, consistent ways covered below.

Managing the connection lifecycle

Open the connection once, when the app starts, and close it once, on shutdown. FastAPI and Express approach this differently — FastAPI has a first-class hook for it; Express has no equivalent, so the convention is simpler and a bit more manual.

main.py
from contextlib import asynccontextmanager
from fastapi import FastAPI
import redis.asyncio as redis

@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.redis = redis.from_url(
        "redis://localhost:6379",
        decode_responses=True,
        max_connections=20,
    )
    yield
    await app.state.redis.aclose()

app = FastAPI(lifespan=lifespan)
server.js
const express = require("express");
const { createClient } = require("redis");

const app = express();
const redisClient = createClient({ url: "redis://localhost:6379" });
redisClient.on("error", (err) => console.error("Redis error", err));

async function start() {
  await redisClient.connect();
  app.listen(3000, () => console.log("Listening on http://localhost:3000"));
}
start();

process.on("SIGTERM", async () => {
  await redisClient.quit();
  process.exit(0);
});

FastAPI's lifespan context manager runs its setup code before the app accepts traffic and its teardown code after the app stops accepting it — both guaranteed, in order. Express has no such mechanism: Node apps connect before calling app.listen(), and clean up manually on process signals like SIGTERM — the signal your container orchestrator sends before killing the process, which is what makes a graceful redisClient.quit() possible instead of the connection being dropped mid-request.

Accessing Redis from a request handler

FastAPI's dependency injection system and Express's middleware model solve the same problem — giving a handler access to a shared resource — with genuinely different architectures.

main.py
from fastapi import Depends, Request
import redis.asyncio as redis

async def get_redis(request: Request) -> redis.Redis:
    return request.app.state.redis

@app.get("/products/{product_id}")
async def read_product(product_id: int, r: redis.Redis = Depends(get_redis)):
    cache_key = f"product:{product_id}"
    cached = await r.get(cache_key)
    ...
server.js
app.use((req, res, next) => {
  req.redis = redisClient;
  next();
});

app.get("/products/:productId", async (req, res) => {
  const cacheKey = `product:${req.params.productId}`;
  const cached = await req.redis.get(cacheKey);
  // ...
});

FastAPI's Depends is a formal system: a dependency is just a callable, and app.dependency_overrides lets tests swap it out cleanly (see Testing & Observability). Express has no built-in equivalent — the idiomatic approach is middleware that attaches a shared resource onto req for every request, or simply importing a module-level client directly where it's needed. Both get Redis into your handler; only one of them is a first-class, override-able dependency system.

The core Redis commands you'll actually use

Every client library is a thin wrapper around the same underlying Redis command set. Learning the commands themselves — not just one library's method names — is what lets you read Redis's own documentation and move between languages without relearning caching from scratch.

Strings: GET / SET

SET writes a string value; GET reads it back. EX (expire after N seconds) and NX (only set if the key doesn't already exist) are options on the SET command itself, defined by Redis — every client just exposes them with different calling conventions.

main.py
await r.set("product:1", json.dumps(product), ex=300)
await r.set("lock:product:1", "1", nx=True, ex=5)
value = await r.get("product:1")
server.js
await r.set("product:1", JSON.stringify(product), { EX: 300 });
await r.set("lock:product:1", "1", { NX: true, EX: 5 });
const value = await r.get("product:1");

Python passes ex/nx as keyword arguments; Node's client takes a single options object whose uppercase keys mirror Redis's own command syntax (SET key value EX seconds NX) almost literally.

Deleting & checking existence: DEL / EXISTS

DEL removes a key immediately. EXISTS checks whether a key is present without transferring its value back — cheaper than a GET when a yes/no answer is all you need.

main.py
await r.delete("product:1")
count = await r.exists("product:1")  # 0 or 1
server.js
await r.del("product:1");
const count = await r.exists("product:1"); // 0 or 1

Python's method is named delete, since del is a reserved Python keyword; Node's client can use del directly.

Counters: INCR / INCRBY

INCR atomically increments a key treated as an integer. This matters under concurrency: two instances calling INCR on the same key at the same instant never race or lose an update, because Redis executes each command atomically — unlike reading a value, adding one in your own code, then writing it back.

main.py
await r.incr("views:product:1")
await r.incrby("views:product:1", 5)
server.js
await r.incr("views:product:1");
await r.incrBy("views:product:1", 5);

Note the pattern that repeats throughout: Python stays snake_case (incrby), Node's client camelCases multi-word commands (incrBy).

Expiration: EXPIRE / TTL

SET ... EX sets an expiration at write time. EXPIRE attaches (or changes) an expiration on a key that already exists, independent of writing to it. TTL asks Redis how many seconds remain — -1 means the key exists but never expires, -2 means the key doesn't exist at all.

main.py
await r.expire("product:1", 300)
seconds_left = await r.ttl("product:1")
server.js
await r.expire("product:1", 300);
const secondsLeft = await r.ttl("product:1");

Iterating keys safely: SCAN

KEYS pattern* is tempting but blocks the entire Redis server while it walks the whole keyspace — a real production hazard on a busy instance. SCAN instead walks the keyspace incrementally using a cursor, without blocking other clients. Always prefer it.

main.py
async for key in r.scan_iter(match="views:*"):
    print(key)
server.js
for await (const key of r.scanIterator({ MATCH: "views:*" })) {
  console.log(key);
}

Both wrap Redis's underlying cursor-based SCAN loop into a single iterable, so you never manage the cursor by hand.

Hashes: HSET / HGETALL

A Redis hash stores multiple named fields under one key — a small dict-within-a-key. Useful for grouping related values (a cached payload alongside its own metadata, for instance) without a second round trip or a second key.

main.py
await r.hset("product:1:meta", mapping={"hits": 0, "last_access": time.time()})
meta = await r.hgetall("product:1:meta")
server.js
await r.hSet("product:1:meta", { hits: 0, lastAccess: Date.now() });
const meta = await r.hGetAll("product:1:meta");

Serialization: prefer validated shapes over raw dicts

Caching a plain object as JSON works, but throws away type safety on the way back out. FastAPI's ecosystem has a natural answer in Pydantic; Express has no single standard equivalent, so the honest answer differs by ecosystem.

main.py
from pydantic import BaseModel

class Product(BaseModel):
    id: int
    name: str
    price: float

async def get_product(product_id: int, r: redis.Redis) -> Product:
    cache_key = f"product:{product_id}"
    cached = await r.get(cache_key)
    if cached:
        return Product.model_validate_json(cached)

    product = await db.fetch_product(product_id)
    await r.set(cache_key, product.model_dump_json(), ex=300)
    return product
server.js
async function getProduct(productId, r) {
  const cacheKey = `product:${productId}`;
  const cached = await r.get(cacheKey);
  if (cached) {
    return JSON.parse(cached);
  }

  const product = await db.fetchProduct(productId);
  await r.set(cacheKey, JSON.stringify(product), { EX: 300 });
  return product;
}

Product.model_validate_json(...) gives you runtime validation on the way out of the cache — if the stored shape ever drifts from what Product expects, it raises instead of silently returning a malformed object. Plain JSON.parse gives no such guarantee; if that matters for your data, pair it with a runtime schema library like zod rather than trusting the cached JSON blindly.

Fail open: don't let Redis become a single point of failure

A cache is an optimization, not a correctness requirement. A Redis blip should slow your endpoint down, not take it down.

main.py
async def get_product(product_id: int, r: redis.Redis) -> Product:
    cache_key = f"product:{product_id}"

    try:
        cached = await r.get(cache_key)
    except redis.RedisError:
        cached = None  # fail open — treat a down cache as a miss

    if cached:
        return Product.model_validate_json(cached)

    product = await db.fetch_product(product_id)

    try:
        await r.set(cache_key, product.model_dump_json(), ex=300)
    except redis.RedisError:
        pass  # best-effort — losing this write is fine, the database is still correct

    return product
server.js
async function getProduct(productId, r) {
  const cacheKey = `product:${productId}`;

  let cached = null;
  try {
    cached = await r.get(cacheKey);
  } catch (err) {
    cached = null; // fail open — treat a down cache as a miss
  }

  if (cached) return JSON.parse(cached);

  const product = await db.fetchProduct(productId);

  try {
    await r.set(cacheKey, JSON.stringify(product), { EX: 300 });
  } catch (err) {
    // best-effort — losing this write is fine, the database is still correct
  }

  return product;
}
💡 "Fail open" vs. "fail closed"

Fail open means treating a cache error as a miss and continuing to serve from the source of truth — the right default here, since the cache is never the only copy of the data. Fail closed means rejecting the request outright on a backing-store error — the right default for something like a Redis-backed rate limiter, where "fail open" would silently disable the safeguard.

Connection resilience

A slow or unreachable Redis shouldn't hang a request indefinitely. Bound how long each client will wait, and control how it retries.

main.py
redis.from_url(
    "redis://localhost:6379",
    decode_responses=True,
    socket_connect_timeout=2,   # seconds to establish a connection
    socket_timeout=2,           # seconds to wait for a command's response
    retry_on_timeout=True,
)
server.js
createClient({
  url: "redis://localhost:6379",
  socket: {
    connectTimeout: 2000, // ms to establish a connection
    reconnectStrategy: (retries) => Math.min(retries * 100, 2000),
  },
});

Python exposes a simple boolean (retry_on_timeout) plus flat timeout options. Node's client instead takes a custom reconnectStrategy function, called after each failed attempt, returning how many milliseconds to wait before the next one — more code, but more control over backoff shape. Combined with fail-open above, a stuck Redis degrades latency, not availability.

Recap

  • FastAPI manages the connection via lifespan; Express connects before listen() and cleans up on process signals.
  • FastAPI exposes Redis through a formal, overridable dependency; Express approximates it with middleware or a shared module.
  • EX/NX are Redis-level SET options, not client inventions — every language just calls them differently.
  • Prefer SCAN over KEYS in anything touching production.
  • Wrap cache reads and writes so a Redis outage degrades performance instead of availability.
