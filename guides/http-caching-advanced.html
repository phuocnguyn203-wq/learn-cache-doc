Advanced HTTP Caching

Cache-Control and max-age get you a flat expiration window. Production HTTP caching goes further: conditional requests, per-response variation, and background revalidation — the mechanisms browsers and CDNs actually rely on.

Conditional requests with ETag

An ETag is a fingerprint of the response body. A client that has a cached copy sends it back via If-None-Match; if it still matches, the server returns an empty 304 Not Modified instead of the full payload — saving bandwidth even after the cached copy's freshness window has technically expired.

server.js
const crypto = require("crypto");

app.get("/products/:productId", async (req, res) => {
  const product = await fetchProduct(Number(req.params.productId));
  const body = JSON.stringify(product);
  const etag = crypto.createHash("md5").update(body).digest("hex");

  if (req.headers["if-none-match"] === etag) {
    return res.status(304).end();
  }

  res.set("ETag", etag);
  res.set("Cache-Control", "public, max-age=60, must-revalidate");
  res.json(product);
});

Express lowercases every request header's key for you, so req.headers["if-none-match"] works regardless of how the client capitalized it — one less thing to normalize yourself. must-revalidate tells the cache: once max-age expires, don't keep using this stale copy — go back and check (via the conditional request above) before serving it again.

An alternative validator: Last-Modified

Hashing a full response body costs CPU. If you already track when a resource last changed, a timestamp is a cheaper validator — at the cost of only second-level precision:

server.js
app.get("/products/:productId", async (req, res) => {
  const { product, lastModified } = await fetchProductWithTimestamp(Number(req.params.productId));

  const ifModifiedSince = req.headers["if-modified-since"];
  if (ifModifiedSince && lastModified <= new Date(ifModifiedSince)) {
    return res.status(304).end();
  }

  res.set("Last-Modified", lastModified.toUTCString());
  res.set("Cache-Control", "public, max-age=60");
  res.json(product);
});

A pleasant detail here: JavaScript's built-in Date.prototype.toUTCString() already produces the exact RFC 7231 HTTP-date format this header requires — no extra library needed.

The Vary header: telling caches a response isn't one-size-fits-all

If a response differs based on a request header — Accept-Language, Authorization, a custom feature-flag header — a cache needs to know that, or it will serve one cached copy to everyone regardless of their own request.

server.js
res.set("Vary", "Accept-Language, Authorization");
⚠️ A real failure mode without it

A shared or CDN cache stores the English response for /products/1. Without Vary: Accept-Language, the next request — from a browser asking for French — gets served the cached English body, because as far as the cache knows, there's only one /products/1 response. This is exactly the cache key problem from Learn, just one layer further out, in a cache you don't control directly.

stale-while-revalidate: never block on a refresh

A modern Cache-Control extension: serve the stale copy immediately while refreshing it in the background, instead of making the user wait for a fresh fetch.

server.js
res.set("Cache-Control", "public, max-age=60, stale-while-revalidate=30");

Within the first 60 seconds, serve fresh from cache. For the next 30 seconds after that, the cache serves the stale copy immediately and triggers a revalidation in the background, so the following request gets a fresh one. After 90 seconds total, a request must wait for a real fetch. Support for this directive varies across browsers and CDNs — check your specific CDN's documentation before relying on it.

CDN edge caching: s-maxage and purging

s-maxage works like max-age, but applies only to shared caches (CDNs, reverse proxies) — letting you set a different freshness window for the edge than for individual browsers:

server.js
res.set("Cache-Control", "public, max-age=0, s-maxage=3600");

This says: browsers should revalidate every time, but the CDN can hold this response for an hour — a common pattern that lets the edge absorb the bulk of traffic while browsers still always check in.

📎 Purging is a separate mechanism

Your origin server has no way to reach into a CDN's cache and remove an entry directly — each CDN exposes its own purge API for that (Cloudflare, Fastly, and CloudFront all differ). If your data can change out-of-band and needs to invalidate CDN-cached copies immediately, you need to call that API explicitly; TTL expiration alone won't do it in time.

Recap

  • ETag/If-None-Match and Last-Modified/If-Modified-Since let a cache revalidate cheaply via a 304 instead of re-downloading the full response.
  • Express lowercases request header keys for you, so lookups like req.headers["if-none-match"] just work.
  • Vary tells caches a response differs by request header — omit it and you risk serving the wrong variant to the wrong client.
  • stale-while-revalidate serves stale content immediately while refreshing in the background.
  • s-maxage lets CDNs cache longer (or differently) than browsers do; purging is a separate, CDN-specific API call.
