Redis with Node.js (node-redis)

Redis Concepts covered what Redis is and how it thinks. This guide is the practical, ground-up reference for actually using it from Node.js: the official redis package (node-redis), the way real production codebases structure the client, every command family in actual JavaScript, and the patterns — rate limiting, distributed locks, stampede-safe caching — that show up in almost every Node service that talks to Redis. If you followed Learn through to the end, this is the natural next stop.

🧭 node-redis vs. ioredis

Two clients dominate the Node.js ecosystem. node-redis (the redis package, maintained by Redis Inc.) is the official client — a modern, promise-based API that's what this guide covers, and what the rest of this site's examples already use. ioredis is the other major option, historically favored for its longer track record with Cluster and Sentinel and its built-in auto-pipelining. Both are legitimate, widely-used-in-production choices; if a codebase you're working in already picked one, learn that one — the underlying Redis commands and concepts on this page transfer directly either way.

Installing and creating the client

Terminal
npm install redis

The single most common mistake in Node.js codebases that use Redis: creating a new client per request. A Redis client isn't a lightweight object — it's a persistent TCP connection, and node-redis multiplexes every command from your whole app over one connection. Create it once, at module load time, and import that same instance everywhere you need Redis.

redis.js
const { createClient } = require("redis");

const client = createClient({
  url: process.env.REDIS_URL || "redis://localhost:6379",
});

// required — an unhandled "error" event on an EventEmitter
// crashes the Node process. Redis client errors are not optional to handle.
client.on("error", (err) => {
  console.error("Redis client error:", err);
});

module.exports = { client };

Every other module in the app does const { client } = require("./redis") and gets the same connection — one client, shared, for the lifetime of the process.

Connecting and shutting down cleanly

createClient only constructs the client object — it doesn't open a connection. Call connect() once, before your server starts accepting traffic, and close it once, when the process is asked to stop.

server.js
const express = require("express");
const { client } = require("./redis");

const app = express();
// ... routes ...

async function start() {
  await client.connect();
  app.listen(3000, () => console.log("Listening on http://localhost:3000"));
}
start();

// SIGTERM is what your container orchestrator (Docker, Kubernetes) sends
// before killing the process — this is what makes a graceful close possible
// instead of the connection just being dropped mid-command.
process.on("SIGTERM", async () => {
  await client.quit();
  process.exit(0);
});

client.quit() waits for any commands already in flight to finish, then closes the connection cleanly. There's also client.disconnect(), which closes immediately without waiting — reach for that only when you specifically need an unconditional, immediate teardown (test cleanup, for example), not as your default shutdown path.

Reconnection strategy

Networks blip and Redis restarts happen. reconnectStrategy is called after every failed connection attempt with the number of retries so far, and returns how many milliseconds to wait before the next one — this is where you implement backoff.

redis.js
const client = createClient({
  url: process.env.REDIS_URL || "redis://localhost:6379",
  socket: {
    connectTimeout: 2000, // ms to establish a connection before giving up
    reconnectStrategy: (retries) => {
      if (retries > 10) {
        // stop retrying — surface as a hard error instead of retrying forever
        return new Error("Redis: too many reconnection attempts");
      }
      // exponential backoff, capped at 3s, so a restart doesn't get hammered
      return Math.min(retries * 100, 3000);
    },
  },
});

Returning a plain number means "try again after this many ms." Returning an Error stops node-redis from retrying further and the client emits that error — combine this with failing open in your route handlers so a Redis outage degrades performance, not availability.

Configuring from the environment

Hardcoding redis://localhost:6379 only works locally. The standard is a single REDIS_URL environment variable, which also carries auth and TLS:

.env
# local
REDIS_URL=redis://localhost:6379

# managed Redis with auth (ElastiCache, Redis Cloud, Upstash, ...)
REDIS_URL=redis://default:mypassword@my-redis-host:6379

# TLS — note the "rediss" (two S's) scheme, most managed providers require this
REDIS_URL=rediss://default:mypassword@my-redis-host:6380

createClient({ url: process.env.REDIS_URL }) parses username, password, host, port, and TLS out of that one string — no separate config flags to keep in sync across environments.

The core commands, in node-redis

Every command family from Redis Concepts has a corresponding method here. The naming convention is consistent throughout: multi-word Redis commands become camelCase (HGETALL → hGetAll, ZINCRBY → zIncrBy), and options that Redis itself takes as uppercase flags (EX, NX) are passed as an options object with those same uppercase keys.

Strings

strings.js
await client.set("product:1", JSON.stringify(product));
await client.set("product:1", JSON.stringify(product), { EX: 300 }); // + 5 min TTL
await client.set("lock:product:1", "1", { NX: true, EX: 5 });        // only if absent
await client.setEx("session:abc123", 3600, sessionId);                // SET + EX in one call

const value = await client.get("product:1");
await client.incr("views:product:1");
await client.incrBy("views:product:1", 5);

Hashes

hashes.js
await client.hSet("user:1000", { name: "Ada", plan: "pro", logins: 42 });
const plan = await client.hGet("user:1000", "plan");
const user = await client.hGetAll("user:1000");   // whole hash as a plain object
await client.hIncrBy("user:1000", "logins", 1);
await client.hDel("user:1000", "plan");

Lists

lists.js
await client.lPush("recent:user:1000", "viewed:product:42");
await client.lTrim("recent:user:1000", 0, 99);      // keep only the newest 100
const recent = await client.lRange("recent:user:1000", 0, 9);

await client.lPush("job:queue", JSON.stringify(job));
const next = await client.rPop("job:queue");         // FIFO: push left, pop right
🧭 Blocking commands need their own connection

BLPOP and friends hold the connection open until an item arrives (or the timeout hits). On the shared client, that would stall every other command your app sends in the meantime. Run blocking commands on a dedicated connection, the same way pub/sub does below — const worker = client.duplicate(); await worker.connect(); — so the rest of your app keeps working while it waits.

Sets

sets.js
await client.sAdd("product:42:tags", ["sale", "electronics", "new"]);
const isOnSale = await client.sIsMember("product:42:tags", "sale");
const tags = await client.sMembers("product:42:tags");
const sharedTags = await client.sInter(["product:42:tags", "product:57:tags"]);

Sorted sets

sorted-sets.js
await client.zAdd("leaderboard", [
  { score: 1500, value: "alice" },
  { score: 1800, value: "bob" },
]);
await client.zIncrBy("leaderboard", 50, "alice");
const top10 = await client.zRange("leaderboard", 0, 9, { REV: true }); // highest first
const withScores = await client.zRangeWithScores("leaderboard", 0, 9, { REV: true });
const rank = await client.zRank("leaderboard", "alice");

Batching and transactions: multi()

Chain commands on client.multi() and they're queued client-side, then sent and executed as one atomic MULTI/EXEC block when you call .exec() — matching the guarantee from Redis Concepts: nothing else runs in between.

transaction.js
const results = await client
  .multi()
  .incr("page:views")
  .expire("page:views", 86400)
  .exec();
// results is an array, one entry per queued command, in order

For the check-then-write pattern that needs WATCH, call it before you read the value your decision depends on:

optimistic-lock.js
async function decrementIfInStock(key) {
  await client.watch(key);
  const stock = Number(await client.get(key));

  if (stock <= 0) {
    await client.unwatch();
    return false;
  }

  const result = await client.multi().decr(key).exec();
  // result is null if `key` changed since watch() — someone else got there first
  return result !== null;
}

Pub/Sub: always a duplicated connection

This is the detail that trips people up first: once a connection issues SUBSCRIBE, node-redis puts it into subscriber mode, and a client in subscriber mode can't run ordinary commands anymore. The standard, documented pattern is client.duplicate() — it returns a new client pointed at the same connection settings, ready to become a dedicated subscriber while your original client keeps handling everything else.

pubsub.js
const subscriber = client.duplicate();
await subscriber.connect();

await subscriber.subscribe("cache-invalidations", (message) => {
  console.log("invalidate:", message);
  localCache.delete(message);
});

// publish from the original client (or any other connected client) —
// never from `subscriber` itself, it's dedicated now
await client.publish("cache-invalidations", "product:42");

See Invalidation at Scale for the full picture of why this pattern matters: it's how one instance tells every other instance in a fleet "your local cache of this key is now wrong."

Lua scripts with EVAL

For atomic logic that needs actual branching — which MULTI/EXEC can't do — pass a Lua script directly. KEYS and ARGV inside the script map to the keys and arguments arrays you pass in.

safe-unlock.js
// only delete the lock if it still holds the token we set — otherwise we might
// delete a lock that a *different* client has since legitimately acquired
const UNLOCK_SCRIPT = `
  if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
  else
    return 0
  end
`;

await client.eval(UNLOCK_SCRIPT, {
  keys: ["lock:product:1"],
  arguments: [lockToken],
});

This exact script is what the distributed lock pattern further down uses to release safely.

Beyond a single instance: Cluster

A single Redis instance runs on one machine's RAM and one CPU core for command execution — Cluster mode shards keys across multiple Redis nodes so both memory and throughput scale horizontally. node-redis talks to a cluster through a separate constructor, createCluster, which handles routing each command to the right node automatically:

cluster.js
const { createCluster } = require("redis");

const cluster = createCluster({
  rootNodes: [
    { url: "redis://127.0.0.1:7000" },
    { url: "redis://127.0.0.1:7001" },
    { url: "redis://127.0.0.1:7002" },
  ],
});
cluster.on("error", (err) => console.error("Redis cluster error:", err));
await cluster.connect();

// used exactly like a normal client — the cluster figures out which node owns the key
await cluster.set("product:1", "...");

Once your commands touch cluster instead of client, the API you already learned above doesn't change. What does change: multi-key commands (like MGET across several keys, or a MULTI transaction) only work atomically if every key involved happens to hash to the same cluster node — in practice, this means using "hash tags" — {user:1000}:profile and {user:1000}:settings — the braces force both keys onto the same node — for anything that must be transactional together. Most teams don't reach for Cluster until a single instance's memory or CPU genuinely becomes the bottleneck — it's real added operational complexity, not a default to start with.

Sessions: connect-redis

For an Express app using express-session, storing sessions in Redis instead of memory is what makes sessions survive a restart and work correctly across multiple app instances behind a load balancer. connect-redis is the standard store adapter, built directly on top of the client you already have:

session.js
const session = require("express-session");
const { RedisStore } = require("connect-redis");
const { client } = require("./redis");

app.use(
  session({
    store: new RedisStore({ client, prefix: "sess:" }),
    secret: process.env.SESSION_SECRET,
    resave: false,
    saveUninitialized: false,
    cookie: { maxAge: 1000 * 60 * 60 * 24 }, // 1 day
  })
);

connect-redis's exact import shape has changed across major versions — check which one your package.json has pinned before copying this verbatim.

Production patterns

Fail open

A cache read or write that throws should degrade to "treat it as a miss," not crash the request — a Redis outage should slow your app down, not take it down. Every helper below builds on this shape:

cache-aside.js
async function getOrSet(key, ttlSeconds, loader) {
  let cached = null;
  try {
    cached = await client.get(key);
  } catch (err) {
    console.error("Redis GET failed, falling back to source:", err);
  }
  if (cached !== null) return JSON.parse(cached);

  const value = await loader();

  try {
    await client.set(key, JSON.stringify(value), { EX: ttlSeconds });
  } catch (err) {
    console.error("Redis SET failed, continuing without caching:", err);
  }

  return value;
}

// usage
const product = await getOrSet(`product:${id}`, 300, () => db.fetchProduct(id));

Cache stampede prevention

When a hot key expires, a burst of concurrent requests can all miss at once and all hammer the database to recompute the same value simultaneously. The standard fix: the first request to see a miss takes a short-lived lock and does the work; everyone else either waits briefly and retries the cache, or falls back to the database directly if the lock holder is too slow.

stampede-safe.js
async function getOrSetSafely(key, ttlSeconds, loader) {
  const cached = await client.get(key);
  if (cached !== null) return JSON.parse(cached);

  const lockKey = `lock:${key}`;
  const gotLock = await client.set(lockKey, "1", { NX: true, EX: 10 });

  if (gotLock) {
    try {
      const value = await loader();
      await client.set(key, JSON.stringify(value), { EX: ttlSeconds });
      return value;
    } finally {
      await client.del(lockKey);
    }
  }

  // someone else is already recomputing it — wait briefly, then just re-check the cache
  await new Promise((resolve) => setTimeout(resolve, 100));
  const retry = await client.get(key);
  if (retry !== null) return JSON.parse(retry);

  // lock holder still isn't done — fall back to the source directly rather than block forever
  return loader();
}

This trades a small amount of extra complexity for turning "N requests all hit the database" into "one request hits the database, N-1 wait a beat." See Invalidation at Scale for the pub/sub-driven version of this same idea across many server instances.

Rate limiting

The simplest correct rate limiter is INCR plus an expiring window — atomic by construction, no lock needed:

fixed-window-limiter.js
async function isAllowed(userId, limit = 100, windowSeconds = 60) {
  const key = `ratelimit:${userId}:${Math.floor(Date.now() / 1000 / windowSeconds)}`;
  const count = await client.incr(key);
  if (count === 1) {
    await client.expire(key, windowSeconds); // only set TTL on the first hit in this window
  }
  return count <= limit;
}

This is a fixed window — it resets sharply at each window boundary, which means a burst right at the edge of two windows can briefly let through close to double the limit. A sliding window, built on a sorted set with timestamps as scores, fixes that at the cost of a little more work per request:

sliding-window-limiter.js
async function isAllowedSliding(userId, limit = 100, windowSeconds = 60) {
  const key = `ratelimit:sliding:${userId}`;
  const now = Date.now();
  const windowStart = now - windowSeconds * 1000;

  await client.zRemRangeByScore(key, 0, windowStart); // drop requests older than the window
  const count = await client.zCard(key);

  if (count >= limit) return false;

  await client.zAdd(key, [{ score: now, value: `${now}-${Math.random()}` }]);
  await client.expire(key, windowSeconds);
  return true;
}

Each member needs to be unique even for requests in the same millisecond, hence the random suffix — ZADD silently no-ops on a duplicate member instead of adding a second entry.

Distributed locks

SET key token NX PX ttl is the standard single-instance-Redis lock: it succeeds only if no one else holds the lock, and the TTL guarantees it's eventually released even if the holder crashes before unlocking explicitly. The token matters — it's how you avoid releasing a lock that isn't yours anymore (yours expired, someone else acquired it, and then your original code finally gets around to calling DEL).

lock.js
const crypto = require("crypto");

async function withLock(resource, ttlMs, fn) {
  const token = crypto.randomUUID();
  const acquired = await client.set(`lock:${resource}`, token, { NX: true, PX: ttlMs });
  if (!acquired) throw new Error(`Could not acquire lock: ${resource}`);

  try {
    return await fn();
  } finally {
    // the Lua script from earlier — only delete if we still hold it
    await client.eval(UNLOCK_SCRIPT, { keys: [`lock:${resource}`], arguments: [token] });
  }
}

await withLock("inventory:sku42", 5000, async () => {
  // exclusive section — only one process runs this at a time
});
⚠️ This lock is good enough for almost everything — know its limit

A single-instance lock like this one is correct as long as that one Redis instance stays up. If you're running Redis Sentinel or a similar multi-node setup and need a lock that survives a failover without a brief window where two processes could both believe they hold it, that's the harder problem the Redlock algorithm addresses — and it's genuinely debated among distributed-systems practitioners whether it fully delivers that guarantee under real-world clock behavior. Reach for it only once you've confirmed you actually need cross-failover correctness, not as a default upgrade.

Testing

Redis's own semantics — TTL races, atomic increments, eviction — are exactly the parts a mock is least likely to reproduce faithfully. The Testing & Observability guide covers this in depth; the short version for Node.js: run a real Redis in CI (a Docker container is one line — docker run -p 6379:6379 redis, or a testcontainers-node Redis module if your suite already uses testcontainers for other services) rather than mocking the client. Point REDIS_URL at it and use the exact same client code your app runs in production.

Common pitfalls, checked against everything above

🚫 Checklist before shipping
  • A new client per request. Create one client at module load, reuse it everywhere — see Installing and creating the client.
  • No error listener. An unhandled error event on the client crashes the whole Node process, not just the request that triggered it.
  • Forgetting await client.connect(). Every command will simply hang waiting for a connection that was never opened.
  • Running normal commands on a subscribed connection. Once a client calls subscribe(), give it nothing else to do — duplicate() for everything else.
  • Caching without a TTL. An unbounded cache with noeviction set (the Redis default) will eventually fail every write — see Eviction.
  • KEYS pattern* in request-handling code. Blocks the entire server while it walks the keyspace. Use SCAN/scanIterator instead, always.

Recap

  • One client per process, created once, reused everywhere — never per-request.
  • Multi-word Redis commands map to camelCase methods; Redis's own uppercase options (EX, NX, REV) become an options object with the same keys.
  • multi()/.exec() gives atomic batching; add watch() for safe check-then-write; reach for eval() when you need real branching logic to stay atomic.
  • Pub/sub and blocking list commands both need a duplicate()-d connection, because a client busy waiting can't run anything else.
  • The recurring production patterns — stampede-safe caching, rate limiting, distributed locks — are all built from the same handful of primitives: SET NX, INCR, sorted sets, and Lua scripts for the parts that need branching.
