Redis

Lesson 8 ended on a cliffhanger: everything we built lives inside one server's memory, so if you ran two copies of that app, each one would keep its own separate, private cache. Redis is the fix — the exact same "box of drawers" idea from Learn, just running as its own separate program that every copy of your app can share. No prior Redis knowledge assumed.

What is Redis, really?

Picture your app's cache — that plain object from Lesson 2 — but instead of living inside your app, it lives in its own separate little program next door. Your app doesn't keep the drawers itself anymore; it walks over (over the network) and asks that other program to check a drawer, or put something in one. If you run five copies of your app, all five walk over to the same next-door program — so they all see the same drawers, instead of each having their own.

📖 Redis, in one sentence

Redis is a separate, extremely fast program whose whole job is holding data in memory and answering "what's in this drawer?" and "put this in that drawer" — shared by every part of your system that wants to ask it.

It's just as fast as our in-memory object was, for the same reason — everything sits in RAM. The only real difference is where that RAM lives: not inside your app's own process, but inside a separate process your app talks to over a connection.

Get a Redis server running

Before your app can talk to Redis, an actual Redis server needs to be running somewhere. The npm install redis package you're about to install isn't Redis itself — it's just the phone your app uses to call it. The fastest way to get a real one running on your own computer is Docker:

Terminal
docker run -p 6379:6379 redis

That starts a real Redis server on your machine, listening on Redis's standard port, 6379.

🧒 No Docker?

You can also install Redis directly on your machine — see the official installation instructions for your OS. Docker is just the fastest way to try it without touching your system at all.

Install the Node.js client

Terminal
npm install redis

This installs node-redis, the official Redis client for Node.js — the recommended, most widely used way to talk to Redis from a JavaScript app.

Connect to it

server.js
const { createClient } = require("redis");

const client = createClient({ url: "redis://localhost:6379" });
client.on("error", (err) => console.error("Redis error:", err));

await client.connect();

createClient just builds the client object — it doesn't connect yet. client.connect() is the step that actually opens the connection to the Redis server you started above. Two properties are useful for checking on it later: client.isReady (true once it's connected and able to take commands) and client.isOpen (true whenever the underlying connection is open, even while it's reconnecting). When you're completely done with a connection, close it with client.quit().

Store and fetch a value: SET and GET

server.js
await client.set("greeting", "hello");
const value = await client.get("greeting");
console.log(value); // "hello"

This is the exact same idea as cache["greeting"] = "hello" from Lesson 2 — put something in a labeled drawer, read it back later by that label. The one real difference: this line actually sends a small message over the network to the separate Redis server and waits for its answer, which is why every Redis command needs an await in front of it.

Give it an expiration date, for free

Remember Lesson 4, where we had to save a timestamp and manually check age < TTL_MS on every single read? Redis will do all of that bookkeeping for you — just tell it how many seconds a value should live:

server.js
await client.set("session:abc123", userId, { EX: 3600 }); // gone automatically after 1 hour

EX means "expire after this many seconds." Once that time passes, Redis quietly removes the key on its own — ask for it after that and you'll just get back nothing (null), exactly like a cache miss. No age-checking code required anywhere in your app.

A quick peek beyond plain strings

SET/GET store plain text — great for a single value, but you'll often cache something with a few named fields, like our weather object. Redis has a ready-made shape for exactly that, called a hash — think of it as a mini object living directly inside one Redis key:

server.js
await client.hSet("weather:paris", { temperature: 24, units: "celsius" });
const weather = await client.hGetAll("weather:paris");
console.log(weather); // { temperature: '24', units: 'celsius' }

Redis has several more of these purpose-built shapes beyond strings and hashes — good to know they exist, not something you need to learn right now. SET/GET alone gets you a very long way.

Rewriting our weather cache with Redis

Here's the payoff — the same weather endpoint from Lesson 8, now backed by real Redis instead of a Map living inside our own process:

server.js
const express = require("express");
const { createClient } = require("redis");

const app = express();
const client = createClient({ url: process.env.REDIS_URL || "redis://localhost:6379" });
client.on("error", (err) => console.error("Redis error:", err));

function sleep(ms) {
  return new Promise((resolve) => setTimeout(resolve, ms));
}

app.get("/weather/:city", async (req, res) => {
  const city = req.params.city;
  const units = req.query.units || "celsius";
  const cacheKey = `weather:${city}:${units}`;

  const cached = await client.get(cacheKey);
  if (cached) {
    return res.json(JSON.parse(cached));
  }

  await sleep(3000);
  const temperature = units === "celsius" ? 24 : 75;
  const weather = { city, temperature, units };
  await client.set(cacheKey, JSON.stringify(weather), { EX: 30 });
  res.json(weather);
});

async function start() {
  await client.connect();
  app.listen(3000);
}
start();

Notice what's gone compared to Lesson 8: no Map, no MAX_ITEMS, no manually saved timestamp, no hand-written age check, no eviction code we had to write ourselves. Redis handles expiration — and, separately, what happens when it runs low on memory — on its own; we just ask for a key and get back either a value or nothing.

And unlike Lesson 8's version, this one keeps working correctly even if you run five copies of this server behind a load balancer — every copy reads and writes the exact same shared Redis, instead of five separate private caches quietly disagreeing with each other.

Recap

  • Redis is a separate, shared, very fast program that plays the same "box of drawers" role our in-memory object/Map did — just reachable by every copy of your app at once.
  • docker run -p 6379:6379 redis gets a real Redis server running in seconds to try this yourself.
  • npm install redis installs node-redis, the official client. Build it with createClient(), then await client.connect() before using it.
  • client.set/client.get work just like our plain object did; the { EX: seconds } option gives you TTL for free, with Redis doing the expiration bookkeeping instead of you.
  • Redis stores more than strings — hashes, and several other purpose-built shapes — worth knowing they exist as you grow past the basics here.
