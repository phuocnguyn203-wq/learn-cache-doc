Caching Patterns

Four canonical patterns cover the overwhelming majority of caching architectures you'll encounter in production. Their names are standard vocabulary in system design — here's each one precisely defined, with its trade-offs and real Node.js/Express code.

Cache-Aside (Lazy Loading)

The application is responsible for the cache: it checks the cache first, loads from the source of truth on a miss, and populates the cache itself. The cache never talks to the database — it's purely passive storage the application manages.

server.js
async function getProduct(productId) {
  const cacheKey = `product:${productId}`;
  const cached = await redis.get(cacheKey);
  if (cached) {
    return JSON.parse(cached);
  }

  const product = await db.fetchProduct(productId);
  await redis.set(cacheKey, JSON.stringify(product), { EX: 300 });
  return product;
}
ProsSimple to reason about; only requested data ever gets cached, so there's no wasted memory on unused keys; the app can degrade gracefully if the cache is unavailable.
ConsEvery key's first request is always a miss ("cold cache" penalty); a very popular key expiring can cause a stampede (see Invalidation at Scale); the cache and database are written separately, so there's a brief window where they can disagree.

Read-Through

The cache sits in front of the data store as a layer the application always talks to. On a miss, the cache itself — not your endpoint code — loads from the source of truth and populates itself. From the call site, there's no explicit "miss" branch to write.

server.js
class ProductCache {
  constructor(redis, loader) {
    this.redis = redis;
    this.loader = loader;
  }

  async get(productId) {
    const cacheKey = `product:${productId}`;
    const cached = await this.redis.get(cacheKey);
    if (cached) {
      return JSON.parse(cached);
    }

    const value = await this.loader(productId);
    await this.redis.set(cacheKey, JSON.stringify(value), { EX: 300 });
    return value;
  }
}

const productCache = new ProductCache(redis, db.fetchProduct);

app.get("/products/:productId", async (req, res) => {
  res.json(await productCache.get(Number(req.params.productId)));
});

Notice the mechanics are nearly identical to cache-aside — the difference is architectural, not functional: who owns the check-load-populate responsibility. Here it's centralized in one class instead of duplicated at every call site. The middleware-based libraries covered next are read-through implementations — your endpoint never sees a miss at all.

ProsOne place owns the caching logic, so behavior is consistent everywhere it's used; far less boilerplate duplicated across endpoints.
ConsLess flexibility per call site; the cache becomes a harder dependency to bypass; still exposed to the same cold-start and stampede issues as cache-aside.

Write-Through

Every write updates the cache and the underlying store together, before the write is acknowledged. Reads immediately after a write always see fresh data.

server.js
app.put("/products/:productId", async (req, res) => {
  const productId = Number(req.params.productId);
  await db.saveProduct(productId, req.body);
  const cacheKey = `product:${productId}`;
  await redis.set(cacheKey, JSON.stringify(req.body), { EX: 300 });
  res.json({ status: "updated" });
});
ProsCache and database stay consistent; no window where a fresh read serves stale data right after a write.
ConsAdded latency on every write, since two systems must be updated; if the cache write fails after the database write succeeds, you get a (usually brief) inconsistency — many teams simply delete the key instead of setting it, falling back to cache-aside on the next read (see Invalidation).

Write-Behind (Write-Back)

Writes land in the cache first and return immediately; the cache persists to the underlying store later, often batched. This is the pattern for write-heavy, loss-tolerant data — view counters, analytics events, telemetry.

server.js
app.post("/products/:productId/views", async (req, res) => {
  await redis.incr(`views:${req.params.productId}`);
  res.json({ status: "recorded" });
});

// a separate periodic worker (node-cron, a queue consumer, a cron job, ...)
async function flushViewCounts() {
  for await (const key of redis.scanIterator({ MATCH: "views:*" })) {
    const count = await redis.get(key);
    const productId = Number(key.split(":")[1]);
    await db.incrementViewCount(productId, Number(count));
    await redis.del(key);
  }
}
ProsLowest possible write latency; many writes can be batched into far fewer database operations, dramatically reducing load for high-frequency data.
ConsData written to the cache but not yet flushed is lost if the cache crashes; requires a reliable flush mechanism (a scheduler, queue, or stream). Never use it for data that needs strong durability — payments, inventory, anything audited.

Choosing a pattern

PatternConsistencyWrite latencyTypical use case
Cache-AsideEventual (bounded by TTL)NormalDefault choice for read-heavy APIs
Read-ThroughEventual (bounded by TTL)NormalSame as cache-aside, centralized behind a library or proxy
Write-ThroughStrong, immediately after writeHigher (dual write)Data that must be correct the instant it's read back
Write-BehindEventual, write-optimizedLowestHigh-frequency writes that tolerate occasional loss
💡 A naming note

"Write-behind" and "write-back" refer to the same pattern — terminology varies by vendor and by which database or caching product's docs you're reading. Recognizing both names will save you a double-take the first time you hit the other one.
