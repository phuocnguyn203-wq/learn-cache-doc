Caching Libraries

Hand-writing "check the cache, load on a miss, populate it" at every endpoint gets repetitive fast. apicache is a real, widely-used library that turns this into a single line added to a route. It's a read-through cache (see Caching Patterns) — your handler doesn't know or care whether it hit or missed.

Installing

Terminal
npm install apicache redis

Initializing

apicache caches in memory automatically by default, and only needs configuring if you want it backed by Redis instead — which you almost always do, the moment you're running more than one server instance (see Redis Concepts and Redis with Node.js).

server.js
const express = require("express");
const apicache = require("apicache");
const { createClient } = require("redis");

const app = express();
const redisClient = createClient({ url: "redis://localhost:6379" });

async function start() {
  await redisClient.connect();
  apicache.options({ redisClient });   // omit this line to use apicache's in-memory store instead
  app.listen(3000);
}
start();

const cache = apicache.middleware;
⚠️ Know the library's default

Because apicache silently falls back to an in-memory store, a misconfigured or never-connected redisClient won't error — it'll just quietly cache per-process instead of per-fleet, reintroducing the exact multi-instance problem Redis exists to solve. Verify the Redis connection explicitly if you're relying on it.

Caching a route

server.js
app.get("/products/:productId", cache("1 minute"), async (req, res) => {
  const product = await db.fetchProduct(req.params.productId);
  res.json(product);
});

cache(...) is literal middleware sitting in the route's handler chain, ahead of your function — on a hit, it responds immediately and your handler never runs at all. That's what "read-through" means in practice: there's no miss branch for you to write.

Customizing the cache key

By default, apicache builds its cache key from the request URL — which doesn't distinguish per-user data by itself. This is exactly the cache key principle from Learn: anything that changes the correct answer must be part of the key.

server.js
apicache.options({
  appendKey: (req, res) => (req.user ? req.user.id : "anonymous"),
});

app.get("/me/orders", cache("30 seconds"), async (req, res) => {
  res.json(await db.fetchOrdersFor(req.user.id));
});

appendKey extends the URL-based default with whatever you return — here, the current user's ID — so /me/orders for one user no longer collides with the cached copy for another.

Manual invalidation

server.js
app.put("/products/:productId", async (req, res) => {
  await db.saveProduct(req.params.productId, req.body);
  apicache.clear(`/products/${req.params.productId}`);
  res.json({ status: "updated" });
});

apicache.clear(...) targets one specific cached URL — precise, and usually exactly what you want after a write. Call it with no argument at all to clear the entire cache instead.

When to reach for a library vs. rolling your own

Use a libraryRead-heavy endpoints you can cache as a whole response, with a uniform TTL and no need for fine-grained, per-key invalidation.
Roll your own (Redis)You need precise per-key invalidation, per-request TTL logic, stampede protection, or caching only part of a response.

Recap

  • apicache implements read-through caching — your handler never sees a miss branch, since the middleware responds on a hit before your function runs.
  • It defaults to caching whole responses in memory; pass a connected redisClient via apicache.options(...) to share the cache across instances.
  • Override the default cache key with appendKey whenever the URL alone wouldn't correctly distinguish two different requests.
  • apicache silently defaults to in-memory storage; verify your Redis wiring explicitly if that's not what you want.
