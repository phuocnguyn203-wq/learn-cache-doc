Testing & Observability

Caching bugs hide well: a wrong TTL, a bad cache key, or a stale value often only surfaces after real time passes in production. Here's how to make that deterministic in tests, and how to know whether a cache is actually earning its complexity once it's live.

Testing cache logic without a real Redis

The two ecosystems solve this with genuinely different philosophies. Python's fakeredis mocks the Redis protocol entirely in memory — no Docker, no real server. The Node.js client ecosystem doesn't have as mature an in-memory fake for the modern redis package, so the common approach instead is testcontainers: spinning up a real, ephemeral Redis in Docker just for the test run.

Terminal
pip install fakeredis
Terminal
npm install --save-dev testcontainers supertest
test_products.py
import fakeredis
import pytest
from fastapi.testclient import TestClient
from main import app, get_redis

@pytest.fixture
def fake_redis():
    return fakeredis.aioredis.FakeRedis(decode_responses=True)

@pytest.fixture
def client(fake_redis):
    app.dependency_overrides[get_redis] = lambda: fake_redis
    yield TestClient(app)
    app.dependency_overrides.clear()

def test_second_request_is_a_cache_hit(client):
    first = client.get("/products/1")
    second = client.get("/products/1")
    assert first.json()["price"] == second.json()["price"]
products.test.js
const { GenericContainer } = require("testcontainers");
const { createClient } = require("redis");
const supertest = require("supertest");
const { createApp } = require("../server");

let container, redisClient, app;

beforeAll(async () => {
  container = await new GenericContainer("redis:7").withExposedPorts(6379).start();
  redisClient = createClient({
    url: `redis://${container.getHost()}:${container.getMappedPort(6379)}`,
  });
  await redisClient.connect();
  app = createApp(redisClient); // server.js exports a factory, not a bound app
});

afterAll(async () => {
  await redisClient.quit();
  await container.stop();
});

test("second request is a cache hit", async () => {
  const first = await supertest(app).get("/products/1");
  const second = await supertest(app).get("/products/1");
  expect(first.body.price).toBe(second.body.price);
});

Both tests hinge on being able to swap the real client for a test one. FastAPI's app.dependency_overrides does this natively. Express has no built-in equivalent, so server.js needs to export a createApp(redisClient) factory instead of a single bound app — the caller supplies whichever client it wants, production or test.

Testing TTL expiration without waiting

Testing a 30-second TTL shouldn't mean an actual 31-second wait in your test suite. Both ecosystems solve this by controlling what "now" means to the code under test, rather than letting real time pass.

Terminal
pip install freezegun
Terminal
# jest ships this built in — nothing extra to install
test_products.py
from freezegun import freeze_time

def test_cache_expires_after_ttl(client):
    with freeze_time("2024-01-01 12:00:00"):
        client.get("/products/1")  # populates the cache

    with freeze_time("2024-01-01 12:00:31"):  # 31s later, past a 30s TTL
        response = client.get("/products/1")
        assert response.json()["cached"] is False
products.test.js
test("cache expires after TTL", async () => {
  jest.useFakeTimers().setSystemTime(new Date("2024-01-01T12:00:00Z"));
  await supertest(app).get("/products/1"); // populates the cache

  jest.setSystemTime(new Date("2024-01-01T12:00:31Z")); // 31s later, past a 30s TTL
  const response = await supertest(app).get("/products/1");
  expect(response.body.cached).toBe(false);

  jest.useRealTimers();
});

freeze_time replaces what time.time() returns inside the with block. jest.useFakeTimers() does the equivalent for Date.now()/new Date() — both make a 31-second boundary test run in milliseconds instead of really waiting.

Measuring hit rate in production

Hit rate — hits divided by (hits + misses) — is the single most useful number for knowing whether a cache is worth its complexity. Both ecosystems have a standard Prometheus client for this.

Terminal
pip install prometheus-client
Terminal
npm install prom-client
main.py
from prometheus_client import Counter

cache_hits = Counter("cache_hits_total", "Cache hits", ["endpoint"])
cache_misses = Counter("cache_misses_total", "Cache misses", ["endpoint"])

async def get_product(product_id: int, r: redis.Redis) -> dict:
    cache_key = f"product:{product_id}"
    cached = await r.get(cache_key)

    if cached:
        cache_hits.labels(endpoint="get_product").inc()
        return json.loads(cached)

    cache_misses.labels(endpoint="get_product").inc()
    product = await db.fetch_product(product_id)
    await r.set(cache_key, json.dumps(product), ex=300)
    return product
server.js
const client = require("prom-client");

const cacheHits = new client.Counter({ name: "cache_hits_total", help: "Cache hits", labelNames: ["endpoint"] });
const cacheMisses = new client.Counter({ name: "cache_misses_total", help: "Cache misses", labelNames: ["endpoint"] });

app.get("/products/:productId", async (req, res) => {
  const cacheKey = `product:${req.params.productId}`;
  const cached = await req.redis.get(cacheKey);

  if (cached) {
    cacheHits.labels("get_product").inc();
    return res.json(JSON.parse(cached));
  }

  cacheMisses.labels("get_product").inc();
  const product = await db.fetchProduct(req.params.productId);
  await req.redis.set(cacheKey, JSON.stringify(product), { EX: 300 });
  res.json(product);
});

app.get("/metrics", async (req, res) => {
  res.set("Content-Type", client.register.contentType);
  res.end(await client.register.metrics());
});

prometheus-fastapi-instrumentator wires up /metrics automatically on the Python side; Express needs the small explicit route shown above. Either way, graph cache_hits_total / (cache_hits_total + cache_misses_total) in Grafana. There's no universal "good" hit rate — it depends entirely on your workload's key distribution. What matters is watching for a drop: it usually means a TTL misconfiguration, a cache-key regression (see Cache Keys), or traffic hitting a cold cache after a deploy.

Logging, for targeted debugging

Metrics tell you that hit rate dropped; a debug-level log line with the cache key tells you which request and why. Keep it at debug level, not info — logging every cache decision on a hot path is noisy and can itself become a performance cost at scale. Reach for metrics for aggregate visibility, and turn logging up only while actively investigating.

Recap

  • fakeredis mocks Redis in-memory for Python tests; the Node ecosystem more commonly spins up a real ephemeral Redis with testcontainers instead.
  • An Express app needs to export an app-creating factory to make its Redis client swappable in tests, since there's no built-in dependency-override system like FastAPI's.
  • freezegun and jest's fake timers both make TTL-boundary tests instant and deterministic.
  • Hit rate is the key production metric in both ecosystems; watch for drops, not absolute thresholds.
