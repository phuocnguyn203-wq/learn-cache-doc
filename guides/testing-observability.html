Testing & Observability

Caching bugs hide well: a wrong TTL, a bad cache key, or a stale value often only surfaces after real time passes in production. Here's how to make that deterministic in tests, and how to know whether a cache is actually earning its complexity once it's live.

Testing cache logic without faking Redis's behavior

Mocking the Redis client is tempting, but Redis's actual behavior — atomic increments, TTL expiration timing, eviction — is exactly what a hand-rolled mock is least likely to get right. The standard, more trustworthy approach is testcontainers: spin up a real, ephemeral Redis in Docker just for the test run, and point your app at that.

Terminal
npm install --save-dev testcontainers supertest
products.test.js
const { GenericContainer } = require("testcontainers");
const { createClient } = require("redis");
const supertest = require("supertest");
const { createApp } = require("../server");

let container, redisClient, app;

beforeAll(async () => {
  container = await new GenericContainer("redis:7").withExposedPorts(6379).start();
  redisClient = createClient({
    url: `redis://${container.getHost()}:${container.getMappedPort(6379)}`,
  });
  await redisClient.connect();
  app = createApp(redisClient); // server.js exports a factory, not a bound app
});

afterAll(async () => {
  await redisClient.quit();
  await container.stop();
});

test("second request is a cache hit", async () => {
  const first = await supertest(app).get("/products/1");
  const second = await supertest(app).get("/products/1");
  expect(first.body.price).toBe(second.body.price);
});

The key design choice that makes this possible: server.js exports a createApp(redisClient) factory instead of one single, already-bound app. That way the caller — your real entrypoint, or this test file — supplies whichever client it wants, production or test, instead of the app reaching for a hardcoded global connection.

Testing TTL expiration without waiting

Testing a 30-second TTL shouldn't mean an actual 31-second wait in your test suite. Jest ships fake timers built in, which let you control what Date.now()/new Date() return inside the code under test, instead of letting real time pass.

products.test.js
test("cache expires after TTL", async () => {
  jest.useFakeTimers().setSystemTime(new Date("2024-01-01T12:00:00Z"));
  await supertest(app).get("/products/1"); // populates the cache

  jest.setSystemTime(new Date("2024-01-01T12:00:31Z")); // 31s later, past a 30s TTL
  const response = await supertest(app).get("/products/1");
  expect(response.body.cached).toBe(false);

  jest.useRealTimers();
});

jest.useFakeTimers().setSystemTime(...) pins what "now" means for any code that calls Date.now() or constructs a new Date(), so a 31-second boundary test runs in milliseconds instead of really waiting — and jest.useRealTimers() at the end hands control back for any tests that run after it.

Measuring hit rate in production

Hit rate — hits divided by (hits + misses) — is the single most useful number for knowing whether a cache is worth its complexity. prom-client is the standard Prometheus client for Node.js.

Terminal
npm install prom-client
server.js
const client = require("prom-client");

const cacheHits = new client.Counter({ name: "cache_hits_total", help: "Cache hits", labelNames: ["endpoint"] });
const cacheMisses = new client.Counter({ name: "cache_misses_total", help: "Cache misses", labelNames: ["endpoint"] });

app.get("/products/:productId", async (req, res) => {
  const cacheKey = `product:${req.params.productId}`;
  const cached = await req.redis.get(cacheKey);

  if (cached) {
    cacheHits.labels("get_product").inc();
    return res.json(JSON.parse(cached));
  }

  cacheMisses.labels("get_product").inc();
  const product = await db.fetchProduct(req.params.productId);
  await req.redis.set(cacheKey, JSON.stringify(product), { EX: 300 });
  res.json(product);
});

app.get("/metrics", async (req, res) => {
  res.set("Content-Type", client.register.contentType);
  res.end(await client.register.metrics());
});

Graph cache_hits_total / (cache_hits_total + cache_misses_total) in Grafana. There's no universal "good" hit rate — it depends entirely on your workload's key distribution. What matters is watching for a drop: it usually means a TTL misconfiguration, a cache-key regression (see Cache Keys), or traffic hitting a cold cache after a deploy.

Logging, for targeted debugging

Metrics tell you that hit rate dropped; a debug-level log line with the cache key tells you which request and why. Keep it at debug level, not info — logging every cache decision on a hot path is noisy and can itself become a performance cost at scale. Reach for metrics for aggregate visibility, and turn logging up only while actively investigating.

Recap

  • testcontainers spins up a real, ephemeral Redis in Docker for tests — more trustworthy than mocking Redis's own behavior by hand.
  • Exporting an app-creating factory (createApp(redisClient)) instead of one bound app is what makes the Redis client swappable between production and tests.
  • Jest's fake timers make TTL-boundary tests instant and deterministic — no real waiting.
  • Hit rate is the key production metric to watch; care about drops, not absolute thresholds.
